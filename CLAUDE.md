# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Development workflow

Every feature or change follows this workflow — keep it in mind from the first line of code:

**When in doubt, ask.** Any uncertainty about *what* to build — the requirement, the desired behaviour, the functional scope — must be resolved by asking the user before implementing. Never guess at intent. *Technical* decisions (architecture, patterns, data structures, naming, test design, refactors), by contrast, are made autonomously to the highest programming standards — do not ask about those, just apply best practice. In short: ask about the functional/feature side, decide the engineering side yourself.

1. **Plan deliberately.** Design so the software stays *extensible* (new cases slot in without rewrites) and *well testable* (logic separable from Android/UI, behaviour reachable by a test). Prefer a reusable building block over a one-off when the same need will recur.
2. **Implement** the plan.
3. **Remove dead code.** Delete now-unused sections, wrappers, imports, and helpers so the codebase stays a clear picture of what's actually in use — no leftovers from the previous shape of the code.
4. **Validate** the change against the requirement (and edge cases) by reading it back.
5. **Write tests** covering the new behaviour (ViewModel logic → JVM unit tests under `app/src/test/`; UI behaviour → Compose tests under `app/src/androidTest/`).
6. **Compile and test from the command line** using the commands below — `assembleDebug` + `testDebugUnitTest` (and `connectedDebugAndroidTest` for instrumented tests when a device/emulator is attached). Don't consider a step done until it builds and tests pass.
7. **Update the docs** — keep this `CLAUDE.md` in sync with new architecture, design decisions, and conventions.
8. **Commit** with a clear, conventional message.

### Backup & data portability

- **Auto Backup**: `backup_rules.xml` (pre-12) and `data_extraction_rules.xml` (Android 12+) explicitly include `trainist.db` + WAL files. Android backs these up to Google Drive automatically when the user has backup enabled — no code required.
- **Full Backup Export** (`data/backup/BackupExporter`): reads all data types — the four nutrition ones (food entries, weight entries, daily goal, custom foods) **plus the training ones** (custom exercises, templates + their exercise slots, sessions + their exercises + sets) — and writes a ZIP to `cacheDir` with one named CSV per section (`food_entries.csv`, `weight_entries.csv`, `daily_goal.csv`, `custom_foods.csv`, `exercises.csv`, `workout_templates.csv`, `template_exercises.csv`, `workout_sessions.csv`, `session_exercises.csv`, `set_entries.csv`), shared via `FileProvider` + `Intent.ACTION_SEND`. Only **custom** exercises are exported — catalog entries are reproduced by the bundled-snapshot seeding on every install, so a referenced catalog exercise resolves against the target's seeded catalog by its stable id rather than travelling in the backup.
- **Full Backup Import** (`data/backup/BackupImporter`): opens a user-picked URI, auto-detects ZIP vs single CSV by checking the PK magic bytes (`0x50 0x4B`). ZIP imports every known section by entry name (unknown/missing entries ignored). A single CSV file is routed by type detection (`detectCsvType`) based on unique column presence — see table below. `Result` reports counts for each data type (incl. `exercisesImported`/`templatesImported`/`sessionsImported`). The Android IO (`import(uri)`, `isZip`) is kept thin; the actual dispatch/parse/save lives in the Android-free top-level `importZipEntries(InputStream, ImportTargets)` and `importCsvLines(List<String>, ImportTargets)` (`ImportTargets` bundles all seven repositories), which are unit-tested end-to-end (incl. legacy nutrition-only backups, a full training round-trip — asserting each session's `templateStableId` link survives, so the templates list's "last used" hint still resolves after a restore — and a fresh-device relink) in `BackupImporterTest`.
- **Relational training data — stable-key reassembly** (`data/backup/WorkoutBackup.kt`, pure/Android-free, `WorkoutBackupTest`): training data is a graph the repositories only persist whole, so the ZIP splits it into flat, name-based CSVs whose **foreign keys are the stable string keys** (`Exercise.stableId`, `WorkoutTemplate.stableId`, `WorkoutSession.stableId`), **never** row ids. On import `WorkoutBackup.assembleTemplates`/`assembleSessions` re-group the flat rows back into domain graphs — `set_entries` rows link to their parent `SessionExercise` by `(session_stable_id, exercise_position)` (position is unique per session, so a duplicated exercise keeps its own sets), and rows are sorted by `position` regardless of file order — then each reassembled `WorkoutTemplate`/`WorkoutSession` is saved fresh (`id = 0`, so the target mints its own row ids). Custom exercises are `upsertAll`-ed first (keyed by `stableId`), so any template/session referencing a custom exercise resolves even on a freshly installed device. The relational sections only reassemble as a whole ZIP; a lone `exercises.csv` still imports standalone via `importCsvLines`, but the other five training CSVs are recognised-but-no-op on their own.
- **CSV type detection** (`detectCsvType` in `BackupImporter.kt`): identifies a CSV by its unique column. **Training sections are matched first** (`set_entries.csv` also carries a `weight_kg` column, so it must win over the weight-entries rule):
  - `is_custom` → `EXERCISES` (parsed by `ExerciseCsvFormat`)
  - `template_name` → `WORKOUT_TEMPLATES`; `target_sets` → `TEMPLATE_EXERCISES` (`WorkoutTemplateCsvFormat` / `TemplateExerciseCsvFormat`)
  - `is_active` → `WORKOUT_SESSIONS`; `superset_group_id` → `SESSION_EXERCISES`; `exercise_position` → `SET_ENTRIES` (`WorkoutSessionCsvFormat` / `SessionExerciseCsvFormat` / `SetEntryCsvFormat`)
  - `food_name` → `FOOD_ENTRIES` (parsed by `CsvFormat`)
  - `weight_kg` → `WEIGHT_ENTRIES` (parsed by `WeightCsvFormat`)
  - `kcal_per_100g` → `CUSTOM_FOODS` (parsed by `CustomFoodCsvFormat`)
  - `kcal` (without the above) → `DAILY_GOAL` (parsed by `GoalCsvFormat`)
- **CsvFormat** (`data/backup/CsvFormat.kt`): pure Kotlin — food entries. Columns identified by name; missing columns fall back to defaults; extra columns are ignored. `CsvFormat.parseHeaders` is the shared header-parsing function used by all format objects.
- **CustomFoodCsvFormat** (`data/backup/CustomFoodCsvFormat.kt`): pure Kotlin — custom foods. Columns: `name`, `brand`, `kcal_per_100g`, `protein_per_100g`, `carbs_per_100g`, `fat_per_100g`, `sugar_per_100g`, `fiber_per_100g`, `salt_per_100g`, `tag`.
- **Food-entry CSV** (`CsvColumns` / `CsvFormat`): the food-entries CSV carries a `tag` column (the `FoodTag` enum name). Both formats parse the tag via `FoodTag.parse`, so a missing column or unknown value → `FoodTag.NONE` — older backups without the column still import untagged.
- **WeightCsvFormat** (`data/backup/WeightCsvFormat.kt`): pure Kotlin — weight entries. Columns: `date`, `weight_kg`, `timestamp_ms`.
- **GoalCsvFormat** (`data/backup/GoalCsvFormat.kt`): pure Kotlin — single-row daily goal. Columns: `kcal`, `protein_g`, `carbs_g`, `fat_g`, `target_weight_kg`. The target column is optional: missing or blank → `targetWeightKg = null` (older exports parse fine).
- **Training CSV formats** (all pure Kotlin, one per section, mirroring `CsvFormat`/`CustomFoodCsvFormat`):
  - **ExerciseCsvFormat** (`exercises.csv`): custom `Exercise`s. Columns: `exercise_stable_id`, `name`, `type`, `is_custom`, `primary_muscles`, `secondary_muscles`, `equipment`, `mechanic`, `category`, `instructions`, `rest_seconds`. List cells (muscles/instructions) are joined by `|` (one line per row — an embedded newline would break the line-based reader) and CSV-escaped; a missing `is_custom` defaults to `true`.
  - **WorkoutTemplateCsvFormat** (`workout_templates.csv`): template header. Columns: `template_stable_id`, `template_name`.
  - **TemplateExerciseCsvFormat** (`template_exercises.csv`): one slot per row. Columns: `template_stable_id` (FK), `exercise_stable_id`, `position`, `target_sets` (legacy plain count, still written for back-compat), `set_types` (the planned sets in order, `SetType` names joined by `|` — same list-cell convention as `ExerciseCsvFormat`). Older backups without `set_types` fall back to `target_sets` NORMAL sets, same as the DB migration's fallback.
  - **WorkoutSessionCsvFormat** (`workout_sessions.csv`): session header. Columns: `session_stable_id`, `date`, `is_active`, `started_at_ms`, `ended_at_ms`, `note`, `template_stable_id`. The last is optional (the template the session was started from, or blank for a free workout) — missing on older backups, which simply import with no known template.
  - **SessionExerciseCsvFormat** (`session_exercises.csv`): one exercise per row. Columns: `session_stable_id` (FK), `exercise_stable_id`, `position`, `superset_group_id`.
  - **SetEntryCsvFormat** (`set_entries.csv`): one set per row. Columns: `session_stable_id` + `exercise_position` (the FK pair to its `SessionExercise`), `position`, `weight_kg`, `reps`, `type`, `completed`.
- **FileProvider** authority: `dev.antonlammers.trainist.fileprovider`, paths configured in `res/xml/file_paths.xml` (cache dir).

### Backup schema evolution — backward compatibility contract

All CSV parsing is name-based (column order is irrelevant on import). This gives us forward and backward compatibility for free, as long as the following rules are followed:

| Change | Export (`toRow` / `HEADER`) | Import (`fromRow`) |
|---|---|---|
| **Add a column** | Add constant to `CsvColumns` / format object, add to `HEADER` and `toRow` | Read with a sensible default for missing values — old exports that lack the column still parse |
| **Remove a column** | Remove from `HEADER` and `toRow` | Remove the read; `fromRow` ignores unknown columns, so new exports with the column dropped still parse in old app versions |
| **Rename a column** | Write both old and new names in `toRow` for at least one release, then drop the old name | Read the new name with fallback to the old name until the transition period ends |
| **Add a new data type** | Add a new `*CsvFormat` object + ZIP entry name in `BackupExporter` | Add a new `CsvType` variant + detection rule in `detectCsvType` (place a more-specific rule before any that shares a column), handle the entry in `importZipEntries` and add a `when` branch in `importCsvLines` |
| **Add a relational data type** | Split it into flat CSVs whose foreign keys are **stable string keys** (never row ids); export a section per table | Collect the sections and reassemble the graph in a pure helper (like `WorkoutBackup`), saving each root with `id = 0` so the target mints row ids; keep it a whole-ZIP operation (component CSVs are no-ops standalone) |
| **Change the ZIP container format** | — | Extend `isZip` / add a new container detection path; keep old paths for legacy files |

**No explicit version field is needed.** Structural detection (column presence, ZIP magic bytes, entry names) is sufficient and avoids the overhead of managing version numbers. Only introduce a version field if two formats become structurally ambiguous (same unique columns, different semantics).

**Referential integrity across devices.** Relational backups (training data) never serialise auto-increment row ids — all cross-row references use the stable string keys, so a backup rebuilds correctly on a freshly installed device whose row ids differ. Referenced custom exercises are created on import (via `exercises.csv`); referenced catalog exercises resolve against the target's seeded catalog.

### Key design decisions

- **Domain layer is Android-free.** ViewModels depend only on `domain/repository` interfaces. Swapping the data source touches only `data/` and `di/`.
- **Bottom navigation — four tabs** (`AppNavigation`, `bottomNavItems`): **Ernährung** (`Screen.Overview`, `overview` — the daily food log/overview), **Training** (`Screen.Workout`, `workout` — the workout-template list), **Statistik** (`Screen.Stats`, `stats`), **Einstellungen** (`Screen.Settings`, `settings`). Icons active = `Icons.Rounded.*`, inactive = `Icons.Outlined.*`: Ernährung = `Restaurant` (cutlery — food, not a house), Training = `FitnessCenter`, Statistik = `BarChart`, Einstellungen = `Settings`. The nav is **icon-only** — `NavigationBarItem` has no `label` (the pictograms are self-explanatory; the German label string lives on as each icon's `contentDescription` for accessibility). There is **no** standalone Goals tab — goal editing lives in the Settings hub. Any `Screen` route not in `bottomNavItems` (`AddFood`, `BarcodeScanner`, `ExerciseCatalog`, `ExerciseDetail`, `TemplateEditor`, `WorkoutSession`, `WorkoutHistory`) hides the bar.
- **Theme isolation — "Ink & Paper".** Replacing `ui/theme/Color.kt`, `Theme.kt`, and `Type.kt` is sufficient to restyle the entire app. The app ships a **fixed monochrome color system** (no Material You): dark mode = "Ink", light mode = "Paper", chosen automatically from the system dark/light setting (`isSystemInDarkTheme()`); there is no in-app toggle. `dynamicColor` defaults to **false** in `TrainistTheme` (kept only for previews/tests). Colour is reserved for data. `Color.kt` holds three token groups: the `Ink*` dark palette (`InkPaper`/`InkBg`/`InkSurface`/`InkSurface2`/`InkLine`/`InkText`/`InkText2`/`InkText3`), the `Paper*` light palette (same roles), and the mode-independent `AccentLilac` (`#B79FCB`, fixed `primary`/`secondary`) + `OnAccentInk` (`#1B1409`, on-accent dark ink), plus the data colours: pastel macros (`CalorieColor`, `ProteinColor`, `CarbsColor`, `FatColor`) and clean-eating tags (`TagHealthyColor`, `TagNeutralColor`, `TagUnhealthyColor`). `Theme.kt` maps these onto the Material3 `darkColorScheme`/`lightColorScheme` roles (`surfaceContainerLowest` = paper/chrome; `outlineVariant` = hairline `line`; `error` = unhealthy tag tone). The base XML `Theme.Trainist` is DayNight-aware: `res/values/themes.xml` uses a light window background, `res/values-night/themes.xml` a dark one (`android:Theme.Material.NoActionBar`) — without the night variant the light window background bled through as a pale hairline at the seam between the content Scaffold and the bottom nav in dark mode.
- **Typography — three downloadable Google fonts** (`ui/theme/Type.kt`): `NewsreaderSerif` (serif — display/headline/title styles: screen titles, big numbers, section/dialog names), `HankenGroteskSans` (sans — body + `labelLarge` buttons/list items/field values), `JetBrainsMono` (mono — `labelMedium`/`labelSmall` uppercase micro-labels with wide tracking, e.g. `KCAL`, `PROTEIN (G)`, axis/chip labels). Loaded at runtime via the Google Fonts provider (`GoogleFont.Provider` + `res/values/font_certs.xml`, dep `androidx.compose.ui:ui-text-google-fonts`) — no binary font assets bundled. Requires Google Play Services at runtime.
- **Icons — Material Symbols Rounded.** All icon usages use the `Icons.Rounded.*` set (`androidx.compose.material.icons.rounded.*`) for the calmer redesign weight, not `Icons.Filled`/`Icons.Default`. Reach for the Rounded variant when adding a new icon.
- **Numeric input fields — `ui/components/NumericTextField`.** UI design language: *every* field that holds a number uses `NumericTextField`, never a bare `OutlinedTextField`. When such a field gains focus it **selects its whole content**, so a pre-filled value (e.g. the last used amount, the current goal) can be overwritten by simply typing — without deleting the old value first. The composable keeps a `String`-based `value`/`onValueChange` API (callers own the raw text and parse it themselves, accepting comma or period as decimal separators via `normalizeDecimal`); the internal `TextFieldValue` that carries the selection never leaks out. `decimal = true` (default) gives a decimal keyboard, `decimal = false` a whole-number keyboard. `label` is nullable — pass `null` when the caller supplies its own static caption above the field (as the Settings goal editor does); optional `textStyle` (e.g. a serif value style), `leadingIcon` (e.g. a colored accent tick), and `placeholder` (a grey hint shown while empty — used for the session's inline-history "last time" values) slots let a screen restyle the field without reintroducing a raw text field. Behaviour is covered by `NumericTextFieldTest` (Compose UI test in `app/src/androidTest/`). When adding a new numeric field, reach for `NumericTextField` — do not reintroduce raw text fields for numbers.
- **Drag-to-reorder lists — `ui/components/DragReorderColumn`.** The one reordering pattern used app-wide: long-press a row's drag handle (`Icons.Rounded.DragIndicator`) to pick it up — a short `HapticFeedbackType.LongPress` tick confirms the grab — then drag it freely; it settles into place on release. Replaced the previous inconsistent mix of up/down arrow buttons (template editor) and a "Nach oben/unten/Löschen" overflow menu (live session), which read as two different reordering idioms for the same interaction. It renders a plain, non-lazy `Column` (meant to be nested as a single item inside an outer `LazyColumn`, not a scrolling list of its own) and takes `items`/`key`/`onMove(from, to)` plus an `itemContent` slot that receives a `rowModifier` (apply to the row's outer container — carries the drag translation + elevation) and a `dragHandleModifier` (apply only to the handle icon, so the gesture never fights the row's own buttons/text fields). Internally it tracks per-item height via `onGloballyPositioned` and, while dragging, calls `onMove` once per adjacent step every time the dragged row crosses a neighbour's mid-height — so `onMove` fires repeatedly through one continuous gesture rather than once on drop, and the ViewModel side stays a simple adjacent-swap (`TemplateEditorViewModel.moveSlot`, `WorkoutSessionViewModel.moveSet`) with no separate "commit on drop" step. **Each row is wrapped in a `key(itemKey) { … }` block** — this is essential, not cosmetic: without it the non-lazy `Column` identifies children by slot position, so the first swap gives the dragged row's slot a new `itemKey`, restarts its `pointerInput(itemKey)`, and cancels the in-progress gesture (`onDragCancel`) — meaning the drag "let go" after a single step and a row could never travel more than one position per press. With the `key` block the row's whole composition (running drag gesture **and** its placement animation) *moves with the logical item* when the list reorders, so one long-press can carry a row across the entire list (first ↔ last). The dragged row follows the finger via `graphicsLayer { translationY }` and eases its leftover offset back to rest on release (an `Animatable` spring); the **other** rows animate their layout shift on each swap via a private `Modifier.animatePlacement()` (standard `onPlaced` + `offset` spring), so a swap reads as a smooth shift rather than an instant jump — placement animation is disabled on the dragged row so it can't fight the manual finger offset. `key` must be genuinely unique per row (not index-based) since duplicates are allowed (e.g. the same exercise twice in a template) — `TemplateSlot` carries a synthetic negative `id` (`newId()`, mirroring the session's existing client-id pattern) purely for this; `SessionExerciseUi`/`SetEntry` already had stable ids. No new dependency — built on `detectDragGesturesAfterLongPress` + `LocalHapticFeedback` + `onPlaced`/`animateTo`, all stock Compose Foundation/UI/Animation.

- **Rest timer** (spec §3.3 — `notification/RestTimer{Scheduler,AlarmReceiver,Notifier}` + `domain/RestTimer`): after a set is checked off, a rest countdown starts (per-exercise duration via `Exercise.restSeconds`, else the global default **180 s**, `RestTimer.DEFAULT_REST_SECONDS`). Unlike the meal reminder's WorkManager job (fine for a once-a-day, imprecise reminder), the alert is scheduled via **`AlarmManager.setAndAllowWhileIdle`** (`RestTimerScheduler.schedule(delayMs)`, `cancel()`), not WorkManager — a rest timer of a couple of minutes needs to reliably fire close to on-time even while the device is idle/locked, which WorkManager's delayed one-time work does not guarantee (Doze can defer it by minutes, which read to the user as "no sound went off"); `setAndAllowWhileIdle` needs no special permission (only the *exact* alarm variants do), and a rest timer doesn't need sub-second precision. `RestTimerAlarmReceiver` is a plain manifest-registered `BroadcastReceiver` (`exported=false`) that just calls `RestTimerNotifier.showExpired`. A natural finish clears the in-app bar **without** a `Cancel`, so the scheduled alert still fires; pause/skip/finish/discard emit `Cancel` (cancels the pending alert *and* dismisses the ongoing notification). The per-exercise override persists via `catalog.upsert` of the exercise with a new `restSeconds`.
  - **Survives leaving/resuming the session**: the rest-timer anchor (`WorkoutSession.restExerciseStableId`/`restTotalSeconds`/`restEndAtMs`/`restPausedRemainingMs`, all null when idle) is persisted onto the session on every start/pause/resume/adjust/cancel (`persistRest`, reusing the same continuous-persistence path as every other session mutation) — so unlike the in-app bar's own view-model-scoped state, the countdown is not lost when the screen is left and re-entered via the templates-list resume banner (a fresh `WorkoutSessionViewModel`). On `init`, an active session's persisted anchor is reconstructed into `_rest` (`restoreRestTimer`) — dropped (and the anchor cleared) if it already ran out while away. The background notification/WorkManager job are untouched by this — they were never tied to the VM's lifetime, so they already survived independently; only the in-app UI state needed a persisted anchor.
- **Workout module** (data model + persistence + exercise catalog & UI — see `docs/workout-spec.md`). Lives parallel to the nutrition side and only touches it at defined points (later: body weight, date). Domain models are Android-free: `Exercise` (catalog + custom), `WorkoutTemplate`/`TemplateExercise`, `WorkoutSession`/`SessionExercise`/`SetEntry`, plus enums `ExerciseType` (WEIGHT_REPS/BODYWEIGHT — additively extensible), `SetType` (WARMUP/NORMAL/DROP/FAILURE), `Mechanic` (COMPOUND/ISOLATION, nullable). Each enum has a defensive `parse` (fallback to a default / null).
  - **Stable string keys.** Everything references an exercise by its `stableId` (catalog = free-exercise-db id, custom = generated UUID), **never** the Room row id — keeps cross-device backup import (a later step) consistent. `WorkoutTemplate`/`WorkoutSession` also carry a `stableId` (UUID) for the same reason. Within the DB, parent→child links (template→its exercises, session→exercises→sets) use row-id foreign keys with `onDelete = CASCADE`.
  - **Session status.** `WorkoutSession.isActive` marks the single in-progress session (persisted continuously so it survives app death / can be resumed); `activeSession()` is the indexed lookup. At most one active session is an app-logic invariant (SQLite can't express a partial unique index Room would validate).
  - **Room shape.** Entities under `data/local/entity/`, nested read POJOs under `data/local/relation/` (`TemplateWithExercises`, `SessionWithExercises` → `SessionExerciseWithSets`, two-level `@Relation`). DAOs are interfaces; `@Transaction @Query` for graph reads. `Exercise` list fields (muscles, instructions) are stored newline-joined in one column and split in mapping (no Room `TypeConverter` introduced). Ordering isn't guaranteed by `@Relation`, so children carry a `position` and mapping sorts by it; on save, positions are re-numbered from list order (canonical).
  - **Repositories.** `ExerciseCatalogRepositoryImpl` (upsert-by-`stableId` via `@Insert(REPLACE)` on the unique index — row-id churn is harmless since nothing references it), `WorkoutTemplateRepositoryImpl`, `WorkoutSessionRepositoryImpl`. Composite writes (a template/session + its children) run inside a single transaction via an injected **`TransactionRunner`** (`RoomTransactionRunner` wraps `db.withTransaction`; a pass-through fake makes the repos unit-testable without Room). Mapping is the pure `WorkoutMappers`.
  - **Exercise catalog snapshot (`data/seed/`).** The bundled catalog is the `free-exercise-db` (yuhonas, **Public Domain**) dataset, shipped **data-only, no images** as a single minified asset `app/src/main/assets/exercise_catalog.json` (~770 KB, 873 exercises; a JSON array with only the fields we use: `id`/`name`/`primaryMuscles`/`secondaryMuscles`/`equipment`/`mechanic`/`category`/`instructions`). `ExerciseSnapshotParser` (pure Moshi, Android-free) parses it into `Exercise`s, deriving `type` from equipment (`"body only"` → `BODYWEIGHT`, else `WEIGHT_REPS`) and marking them non-custom. **No runtime API** — access is only ever through the `ExerciseCatalog` repository interface, so the source is swappable without touching feature code.
  - **Seeding.** `ExerciseCatalogSeeder.seedIfNeeded()` (Android-free; injected `ExerciseCatalogRepository` + `SeedVersionStore` + `ExerciseSnapshotSource`) seeds once per snapshot version: it no-ops when the stored version ≥ `SNAPSHOT_VERSION` (constant — bump it when the asset changes), else `upsertAll`s the parsed snapshot and records the version. Idempotent and custom-preserving because the upsert keys on `stableId` (catalog ids vs. custom UUIDs never collide). Wired in DI as `SharedPrefsSeedVersionStore` (own `trainist_catalog` prefs) + `AssetExerciseSnapshotSource` (reads the asset off the main thread); `TrainistApp.onCreate` launches `seedIfNeeded()` on a background scope.
  - **Training tab structure.** The tab root (`Screen.Workout`) is the **workout-template list** (`TemplatesScreen` + `TemplatesViewModel`) — per spec §2 the Training tab is the "start point" (templates + active session). The exercise catalog is a **sub-screen** (`Screen.ExerciseCatalog`) reached via the templates top-bar library icon; the **template editor** (`Screen.TemplateEditor/{templateId}`, id 0 = new), the **live session** (`Screen.WorkoutSession?templateId={templateId}`, 0 = empty) and the **history** (`Screen.WorkoutHistory`, reached via the templates top-bar calendar icon) are the other sub-screens. All sub-screens have back buttons and are absent from `bottomNavItems`, so the bottom bar hides on them (like `AddFood`).
  - **Templates list** (`ui/workout/TemplatesScreen` + `TemplatesViewModel`): flat hairline template cards (name in serif `titleMedium` + a mono "last used" line — "Heute" / "Gestern" / "Vor N Tagen" / "Noch nie trainiert" — no exercise/set counts, no catalog lookup needed). The top bar has a **history** action (`CalendarMonth` → `Screen.WorkoutHistory`), a **new-template** action (`+`) and an exercise-**catalog** action (`MenuBook`); the accent **Extended FAB "Workout starten"** starts an empty session. Tapping a card **starts a session from that template**; StartToEnd swipe → edit; EndToStart swipe → deferred delete with undo snackbar (same `_pendingDelete` pattern as custom foods/exercises); long-pressing the leading **drag handle** (`DragIndicator`, via `DragReorderColumn` — see the drag-to-reorder bullet) reorders the list manually, persisted immediately. When a session is active, a primary-container **resume banner** ("Laufende Einheit — tippen zum Fortsetzen") sits above the list; `TemplatesViewModel.activeSession` (from `WorkoutSessionRepository.activeSession()`) drives it. Empty state shows a centered `FitnessCenter` hint.
    - **Last used** (drives "what have I not trained in a while"): `WorkoutSession.templateStableId` links a session back to the template it was started from (`null` for a free/empty workout) — set once in `WorkoutSessionViewModel.buildFromTemplate` when the session is created, never touched again. `TemplatesViewModel.templates` is `StateFlow<List<TemplateListItem>>` (`TemplateListItem(template, lastUsedDate: LocalDate?)`), combining `WorkoutTemplateRepository.templates()` with the **full** `WorkoutSessionRepository.sessions()` history to take, per `templateStableId`, the max session `date` (active sessions count too — training with it today counts as "used today"; a template never referenced by any session has a null `lastUsedDate`). The screen renders it via a private `LocalDate?.lastUsedText()`: "Heute" / "Gestern" / "Vor N Tagen" / "Noch nie trainiert" (day-granularity, `ChronoUnit.DAYS` against `LocalDate.now()` — a UI-layer formatter, same pattern as `AddFoodScreen`'s `formatRelative()`).
    - **Manual order** (`WorkoutTemplateRepository.reorder(templateIds)`): unlike template *exercises* or session *sets* (which carry their `position` in the domain model since children are always read already-ordered), the templates table's own row order is a storage-only concern — `WorkoutTemplateEntity.position` never appears on the domain `WorkoutTemplate`. `WorkoutTemplateRepositoryImpl.save` assigns a template's position (existing position preserved via `WorkoutTemplateDao.positionOf` on an edit — the id≠0 REPLACE would otherwise blow it away — else appended via `nextPosition` = current max + 1); `reorder` just writes `0..N-1` over the given id order in one transaction. `TemplatesViewModel.moveTemplate(from, to)` swaps two adjacent templates and calls `reorder` immediately — same "swap once per adjacent drag step, persist continuously" contract as `TemplateEditorViewModel.moveSlot` / `WorkoutSessionViewModel.moveSet`.
  - **Live session** (`ui/workout/WorkoutSessionScreen` + `WorkoutSessionViewModel`, spec §3.3): the single active session is the source of truth (`MutableStateFlow<WorkoutSession?>`) and is **persisted on every change** (each mutation calls `persist()`, serialized by a `Mutex` so the first insert assigns the row id before any follow-up save reuses it → no duplicate rows). On entry the VM **resumes** the existing active session if there is one; otherwise it builds a fresh one (empty, or seeded from a template: each `TemplateExercise` → a `SessionExercise` with one empty set per planned `TemplateExercise.setTypes` entry, each pre-set to its planned `SetType`) and immediately persists it. In-session additions get **synthetic negative client ids** (`newId()`) so each exercise/set has a stable UI key surviving reorder (the mappers auto-generate real row ids, so these are never persisted). The UI (`uiState` = the session joined with `ExerciseCatalogRepository` for each exercise's name + `ExerciseType`, **and** with `sessions()` history for the inline-history hints) is a list of exercise cards; each has editable set rows, rendered via `DragReorderColumn` (see the drag-to-reorder bullet) for drag-handle reordering — a **drag handle**, a **set-type badge** (the set number tinted by its `SetType`, tap → menu), a weight `NumericTextField` (caption "KG", or "ZUSATZ" for `BODYWEIGHT`), a reps `NumericTextField`, a check-off toggle (`CheckCircle`/`RadioButtonUnchecked`), and a delete ✕ — plus "Satz hinzufügen"; a shared `ExercisePickerSheet` adds exercises. Once a work set is logged, each card shows a **VOLUMEN / E1RM summary footer** (mono caption + serif value, hairline-separated) computed by `WorkoutMetrics` (see the training-calculations bullet). **Set-row text state** is local `remember(set.id)` (so intermediate values like "82." survive and it re-seeds only when the stable id changes). Each exercise-card header carries a **rest-duration chip** (per-exercise override menu), and a sticky **`RestTimerBar`** (Scaffold `bottomBar`) appears while a rest countdown runs — mm:ss over a progress line with ±15 s / pause-resume / skip. Top bar: back leaves the session **running** (persisted, resumable); `Fertig` finishes it (`isActive=false`, `endedAtMs=clock()`) — unless the session's **total volume is 0 kg** (no weight/reps logged on any exercise, or no exercises at all), in which case it is deleted outright instead, same as a discard, so empty workouts never clutter the history; the `✕` opens a discard `AlertDialog` (deletes it unconditionally). `finish()`/`discard()` flip a `finished` StateFlow the screen observes to `popBackStack`. The `clock: () -> Long` is injected via a secondary `@Inject` constructor (default `System::currentTimeMillis`) so the primary constructor stays unit-testable with a fixed clock. The VM also depends on `WeightRepository` to resolve the session's body weight (for bodyweight-exercise calculations).
  - **Set types** (`domain/model/SetType`: WARMUP/NORMAL/DROP/FAILURE, NORMAL default): pickable per set via the badge menu; `setSetType` mutates + persists like any other edit. `ui/workout/SetTypeUi.kt` maps each type to an **existing** colour token (no new colours, spec §6): NORMAL = `onSurfaceVariant`, WARMUP = `TagNeutralColor` (amber), DROP = `primary` (accent), FAILURE = `error` — plus `displayName()`/`shortLabel()` (W/D/F).
  - **Inline history** (`domain/InlineHistory` — pure, Android-free, unit-tested like `MacroCalculator`): `lastPerformance(history, exerciseStableId)` returns the ordered set values (`SetPerformance(weightKg, reps)`) of an exercise from the most recent **completed** session containing it (active session ignored); `placeholderForSet(last, i)` picks the value at set index `i`. The session VM feeds each exercise's `lastPerformance` into `uiState`, and the screen shows it as **grey placeholder hints** (`NumericTextField(placeholder = …)`) in empty weight/reps fields — orientation from last time, overwritable by typing.
  - **Training calculations** (`domain/WorkoutMetrics` — pure, Android-free, unit-tested like `MacroCalculator`, spec §3.4/§3.5): the **effective weight** of a set is the entered weight for `WEIGHT_REPS`, and **tracked body weight + added weight** for `BODYWEIGHT` (the real link to the nutrition side). Body weight comes from `resolveBodyWeightKg(entries, date)` = the most recent `WeightEntry` on/before the session date, else the earliest known, else null (→ count only the added weight); the session VM resolves it once from `WeightRepository.allEntries()` and feeds it into `uiState`. **Volume** = Σ (effective weight × reps) with **warm-ups excluded** (`SetType.WARMUP`). **Estimated 1RM** per set = Epley (`weight × (1 + reps/30)`), display-only; the card footer shows the best (highest) Epley over performed sets. **PR detection**: `bestEffectiveWeightKg` (max over non-warm-up, performed (reps ≥ 1) sets) + `isNewPersonalRecord` (strictly-greater, so a tie is not a PR); `personalRecords(completedSessions, typeOf, bodyWeightForDate)` sweeps the whole history chronologically and returns the `(sessionStableId, exerciseStableId)` pairs that set a new max-weight PR (a session keeps its badge even once beaten). The live session surfaces volume + 1RM; the **PR trophy badge** is shown in the **history** day detail (`WorkoutHistoryScreen`, spec §3.5/§3.6).
  - **History** (`ui/workout/WorkoutHistoryScreen` + `WorkoutHistoryViewModel`, spec §3.6): a flat hairline **month calendar** (Monday-first `weeksOf(month)`, mono weekday captions, `ChevronLeft`/`Right` month nav, serif month title) with a discreet **accent dot on training days** and an accent-filled circle on the selected day; tapping a day shows its completed session(s). Each session renders as a card (EINHEIT + total volume, delete icon) whose exercises show the name, a **PR trophy badge** (`EmojiEvents` + mono "PR" in accent, when `personalRecords` flags it), the VOLUMEN/E1RM footer, and **editable set rows** (shared `SetTypeBadge` + weight/reps `NumericTextField`s + a delete-set button) plus "Satz hinzufügen" — so a past unit's sets/weight/reps/type can be corrected in place, and the whole session deleted with an undo snackbar. Editing mirrors the live session's model: the selected day's sessions are held in a **local editable copy** (`_editable`, reseeded only on day change, since `save()` rewrites the set graph and reassigns child ids), mutated + persisted immediately; the calendar markers and PR sweep stay reactive off the repository. `clock` is injected for testable "today"/month; `WorkoutHistoryViewModelTest` covers grouping (training days per month, active session excluded), day selection (volume + PR surfacing), set edits (weight/reps/type/add/remove persist + reindex), and delete/undo/confirm.

- **App versioning — git-derived build counter.** `versionCode`/`versionName` are not hand-maintained: `app/build.gradle.kts`'s `gitCommitCount()` runs `git rev-list --count HEAD` at configure time, so `versionCode = <commit count>` and `versionName = "1.0.<commit count>"` always reflect exactly how many commits are checked out — no file to remember to bump. `buildFeatures.buildConfig = true` exposes it as `BuildConfig.VERSION_NAME`, shown at the bottom of the Settings hub (`SettingsScreen`'s `VersionFooter`). `scripts/git-hooks/prepare-commit-msg` stamps every commit message with a `Version: 1.0.<N>` trailer (`<N>` = the count the new commit will produce, i.e. current `HEAD` count + 1; skipped if the message already carries the trailer, so `--amend` doesn't double it), so the version a commit produced is readable straight from `git log` without a checkout. The hook is tracked at `scripts/git-hooks/prepare-commit-msg`; since it's already installed at `.git/hooks/prepare-commit-msg` in this checkout it needs no further setup here, but **a fresh clone must copy it in manually** (`cp scripts/git-hooks/prepare-commit-msg .git/hooks/` — hooks aren't tracked by git and this repo doesn't set `core.hooksPath`) for the trailer to keep appearing on new commits. **CI checkouts must fetch full history** — `git rev-list --count HEAD` only sees the commits actually present locally, so `actions/checkout@v4`'s default shallow clone (`fetch-depth: 1`) would make every CI-built APK report `1.0.1` regardless of the real commit count; both `.github/workflows/release.yml` and `preview.yml` pass `fetch-depth: 0`.
