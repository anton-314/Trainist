# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Development workflow

Every feature or change follows this workflow — keep it in mind from the first line of code:

**When in doubt, ask.** Any uncertainty about *what* to build — the requirement, the desired behaviour, the functional scope — must be resolved by asking the user before implementing. Never guess at intent. *Technical* decisions (architecture, patterns, data structures, naming, test design, refactors), by contrast, are made autonomously to the highest programming standards — do not ask about those, just apply best practice. In short: ask about the functional/feature side, decide the engineering side yourself.

1. **Plan deliberately.** Design so the software stays *extensible* (new cases slot in without rewrites) and *well testable* (logic separable from Android/UI, behaviour reachable by a test). Prefer a reusable building block over a one-off when the same need will recur.
2. **Implement** the plan.
3. **Remove dead code.** Delete now-unused sections, wrappers, imports, and helpers so the codebase stays a clear picture of what's actually in use — no leftovers from the previous shape of the code.
4. **Validate** the change against the requirement (and edge cases) by reading it back.
5. **Write tests** covering the new behaviour (ViewModel logic → JVM unit tests under `app/src/test/`; UI behaviour → Compose tests under `app/src/androidTest/`).
6. **Compile and test from the command line** using the commands below — `assembleDebug` + `testDebugUnitTest` (and `connectedDebugAndroidTest` for instrumented tests when a device/emulator is attached). Don't consider a step done until it builds and tests pass.
7. **Update the docs** — keep this `CLAUDE.md` in sync with new architecture, design decisions, and conventions.
8. **Commit** with a clear, conventional message.

### Tests

Unit tests live in `app/src/test/`. Fakes (no mocking library) are in `.../fake/`:
- `FakeFoodEntryRepository`, `FakeGoalRepository`, `FakeFoodSearchRepository`, `FakeWeightRepository`, `FakeCustomFoodRepository`
- Workout module: `FakeExerciseCatalogRepository`, `FakeWorkoutTemplateRepository`, `FakeWorkoutSessionRepository` (the session fake encodes the "at most one active session" invariant).

Tests use `kotlinx-coroutines-test` + `turbine`. All ViewModels have full test coverage. `WorkoutViewModelTest` covers the catalog search/filter and custom-exercise CRUD (a `WhileSubscribed` StateFlow only computes real values while collected, so these tests keep it hot with a `backgroundScope` collector and then read `.value`). `ExerciseSnapshotParserTest` covers the snapshot transform (incl. parsing the real bundled asset via `File("src/main/assets/…")`) and `ExerciseCatalogSeederTest` covers the idempotent, custom-preserving seeding with an in-memory `SeedVersionStore` + a counting `ExerciseSnapshotSource`.

Entity↔domain mapping is kept in pure, Android-free helpers so it can be unit-tested without Room (like `CsvFormat`): the workout module's mapping lives in `WorkoutMappers` (tested by `WorkoutMappersTest`). Repository behaviour is tested with in-memory fake DAOs + a pass-through `TransactionRunner` (`WorkoutRepositoriesTest`) — no Robolectric/instrumentation needed for `testDebugUnitTest`.

Compose UI behaviour **and** Room migrations are tested under `app/src/androidTest/` — instrumented, run with `connectedDebugAndroidTest` against an emulator/device, not `testDebugUnitTest`. `WorkoutMigrationTest` exercises the real v7 → v8 migration on Android's SQLite (creates a v7 DB, runs `MIGRATION_7_8`, asserts the new tables + preserved data + FK cascade). Room's schema JSON is exported to `app/schemas/` (`exportSchema = true` + the `room.schemaLocation` KSP arg) — the `MIGRATION_7_8` CREATE statements are copied verbatim from the generated `8.json`, so the migrated schema matches the entities exactly.

### Backup & data portability

- **Auto Backup**: `backup_rules.xml` (pre-12) and `data_extraction_rules.xml` (Android 12+) explicitly include `macrotrac.db` + WAL files. Android backs these up to Google Drive automatically when the user has backup enabled — no code required.
- **Full Backup Export** (`data/backup/BackupExporter`): reads all four data types (food entries, weight entries, daily goal, custom foods), writes a ZIP to `cacheDir` containing four named CSVs (`food_entries.csv`, `weight_entries.csv`, `daily_goal.csv`, `custom_foods.csv`), shares via `FileProvider` + `Intent.ACTION_SEND`.
- **Full Backup Import** (`data/backup/BackupImporter`): opens a user-picked URI, auto-detects ZIP vs single CSV by checking the PK magic bytes (`0x50 0x4B`). ZIP imports all four sections by entry name. A single CSV file is routed by type detection (`detectCsvType`) based on unique column presence — see table below. `Result` reports counts for each data type. The Android IO (`import(uri)`, `isZip`) is kept thin; the actual dispatch/parse/save lives in the Android-free top-level `importZipEntries(InputStream, …repos)` and `importCsvLines(List<String>, …repos)`, which are unit-tested end-to-end (incl. legacy backups from older app versions) in `BackupImporterTest`.
- **CSV type detection** (`detectCsvType` in `BackupImporter.kt`): identifies a CSV by its unique column:
  - `food_name` → `FOOD_ENTRIES` (parsed by `CsvFormat`)
  - `weight_kg` → `WEIGHT_ENTRIES` (parsed by `WeightCsvFormat`)
  - `kcal_per_100g` → `CUSTOM_FOODS` (parsed by `CustomFoodCsvFormat`)
  - `kcal` (without the above) → `DAILY_GOAL` (parsed by `GoalCsvFormat`)
- **CsvFormat** (`data/backup/CsvFormat.kt`): pure Kotlin — food entries. Columns identified by name; missing columns fall back to defaults; extra columns are ignored. `CsvFormat.parseHeaders` is the shared header-parsing function used by all format objects.
- **CustomFoodCsvFormat** (`data/backup/CustomFoodCsvFormat.kt`): pure Kotlin — custom foods. Columns: `name`, `brand`, `kcal_per_100g`, `protein_per_100g`, `carbs_per_100g`, `fat_per_100g`, `sugar_per_100g`, `fiber_per_100g`, `salt_per_100g`, `tag`.
- **Food-entry CSV** (`CsvColumns` / `CsvFormat`): the food-entries CSV carries a `tag` column (the `FoodTag` enum name). Both formats parse the tag via `FoodTag.parse`, so a missing column or unknown value → `FoodTag.NONE` — older backups without the column still import untagged.
- **WeightCsvFormat** (`data/backup/WeightCsvFormat.kt`): pure Kotlin — weight entries. Columns: `date`, `weight_kg`, `timestamp_ms`.
- **GoalCsvFormat** (`data/backup/GoalCsvFormat.kt`): pure Kotlin — single-row daily goal. Columns: `kcal`, `protein_g`, `carbs_g`, `fat_g`, `target_weight_kg`. The target column is optional: missing or blank → `targetWeightKg = null` (older exports parse fine).
- **FileProvider** authority: `dev.antonlammers.macrotrac.fileprovider`, paths configured in `res/xml/file_paths.xml` (cache dir).

### Backup schema evolution — backward compatibility contract

All CSV parsing is name-based (column order is irrelevant on import). This gives us forward and backward compatibility for free, as long as the following rules are followed:

| Change | Export (`toRow` / `HEADER`) | Import (`fromRow`) |
|---|---|---|
| **Add a column** | Add constant to `CsvColumns` / format object, add to `HEADER` and `toRow` | Read with a sensible default for missing values — old exports that lack the column still parse |
| **Remove a column** | Remove from `HEADER` and `toRow` | Remove the read; `fromRow` ignores unknown columns, so new exports with the column dropped still parse in old app versions |
| **Rename a column** | Write both old and new names in `toRow` for at least one release, then drop the old name | Read the new name with fallback to the old name until the transition period ends |
| **Add a new data type** | Add a new `*CsvFormat` object + ZIP entry name in `BackupExporter` | Add a new `CsvType` variant + detection rule in `detectCsvType`, add a `when` branch in `importZip` and `importSingleCsv` |
| **Change the ZIP container format** | — | Extend `isZip` / add a new container detection path; keep old paths for legacy files |

**No explicit version field is needed.** Structural detection (column presence, ZIP magic bytes, entry names) is sufficient and avoids the overhead of managing version numbers. Only introduce a version field if two formats become structurally ambiguous (same unique columns, different semantics).

### Key design decisions

- **Domain layer is Android-free.** ViewModels depend only on `domain/repository` interfaces. Swapping the data source touches only `data/` and `di/`.
- **Bottom navigation — four tabs** (`AppNavigation`, `bottomNavItems`): **Ernährung** (`Screen.Overview`, `overview` — the daily food log/overview), **Training** (`Screen.Workout`, `workout` — the exercise catalog), **Statistik** (`Screen.Stats`, `stats`), **Einstellungen** (`Screen.Settings`, `settings`). Icons active = `Icons.Rounded.*`, inactive = `Icons.Outlined.*`: Ernährung = `Restaurant` (cutlery — food, not a house), Training = `FitnessCenter`, Statistik = `BarChart`, Einstellungen = `Settings`. The nav is **icon-only** — `NavigationBarItem` has no `label` (the pictograms are self-explanatory; the German label string lives on as each icon's `contentDescription` for accessibility). There is **no** standalone Goals tab — goal editing lives in the Settings hub. Any `Screen` route not in `bottomNavItems` (`AddFood`, `BarcodeScanner`) hides the bar.
- **Theme isolation — "Ink & Paper".** Replacing `ui/theme/Color.kt`, `Theme.kt`, and `Type.kt` is sufficient to restyle the entire app. The app ships a **fixed monochrome color system** (no Material You): dark mode = "Ink", light mode = "Paper", chosen automatically from the system dark/light setting (`isSystemInDarkTheme()`); there is no in-app toggle. `dynamicColor` defaults to **false** in `MacroTracTheme` (kept only for previews/tests). Colour is reserved for data. `Color.kt` holds three token groups: the `Ink*` dark palette (`InkPaper`/`InkBg`/`InkSurface`/`InkSurface2`/`InkLine`/`InkText`/`InkText2`/`InkText3`), the `Paper*` light palette (same roles), and the mode-independent `AccentLilac` (`#B79FCB`, fixed `primary`/`secondary`) + `OnAccentInk` (`#1B1409`, on-accent dark ink), plus the data colours: pastel macros (`CalorieColor`, `ProteinColor`, `CarbsColor`, `FatColor`) and clean-eating tags (`TagHealthyColor`, `TagNeutralColor`, `TagUnhealthyColor`). `Theme.kt` maps these onto the Material3 `darkColorScheme`/`lightColorScheme` roles (`surfaceContainerLowest` = paper/chrome; `outlineVariant` = hairline `line`; `error` = unhealthy tag tone). The base XML `Theme.MacroTrac` is DayNight-aware: `res/values/themes.xml` uses a light window background, `res/values-night/themes.xml` a dark one (`android:Theme.Material.NoActionBar`) — without the night variant the light window background bled through as a pale hairline at the seam between the content Scaffold and the bottom nav in dark mode.
- **Typography — three downloadable Google fonts** (`ui/theme/Type.kt`): `NewsreaderSerif` (serif — display/headline/title styles: screen titles, big numbers, section/dialog names), `HankenGroteskSans` (sans — body + `labelLarge` buttons/list items/field values), `JetBrainsMono` (mono — `labelMedium`/`labelSmall` uppercase micro-labels with wide tracking, e.g. `KCAL`, `PROTEIN (G)`, axis/chip labels). Loaded at runtime via the Google Fonts provider (`GoogleFont.Provider` + `res/values/font_certs.xml`, dep `androidx.compose.ui:ui-text-google-fonts`) — no binary font assets bundled. Requires Google Play Services at runtime.
- **Icons — Material Symbols Rounded.** All icon usages use the `Icons.Rounded.*` set (`androidx.compose.material.icons.rounded.*`) for the calmer redesign weight, not `Icons.Filled`/`Icons.Default`. Reach for the Rounded variant when adding a new icon.
- **Numeric input fields — `ui/components/NumericTextField`.** UI design language: *every* field that holds a number uses `NumericTextField`, never a bare `OutlinedTextField`. When such a field gains focus it **selects its whole content**, so a pre-filled value (e.g. the last used amount, the current goal) can be overwritten by simply typing — without deleting the old value first. The composable keeps a `String`-based `value`/`onValueChange` API (callers own the raw text and parse it themselves, accepting comma or period as decimal separators via `normalizeDecimal`); the internal `TextFieldValue` that carries the selection never leaks out. `decimal = true` (default) gives a decimal keyboard, `decimal = false` a whole-number keyboard. `label` is nullable — pass `null` when the caller supplies its own static caption above the field (as the Settings goal editor does); optional `textStyle` (e.g. a serif value style) and `leadingIcon` (e.g. a colored accent tick) slots let a screen restyle the field without reintroducing a raw text field. Behaviour is covered by `NumericTextFieldTest` (Compose UI test in `app/src/androidTest/`). When adding a new numeric field, reach for `NumericTextField` — do not reintroduce raw text fields for numbers.
- **Settings hub** (`ui/settings/SettingsScreen`): the "Einstellungen" tab combines the daily-goals editor and the "Daten" section (backup Export/Import + reminder toggle) in one scrollable screen; it co-hosts `GoalsViewModel` and `DataViewModel`. As a top-level tab it has no back button — the goal editor's "Speichern" saves and shows a snackbar ("Ziele gespeichert") instead of popping back. `GoalsSection`/`DataSection` are `ColumnScope` extension composables so their `Modifier.align` children work. `GoalsViewModel` still owns the goal logic; only its UI moved here (the former `GoalsScreen` was removed).
  - **Goals editor** (`GoalsSection`): each input is a `GoalField` — a mono-uppercase static caption (`FieldLabel`) above a serif-valued `NumericTextField` (`label = null`), with an optional 8×20dp `AccentTick` in the field's leading slot carrying the macro's data color (Protein/Fett/Kohlenhydrate/Kalorien → `ProteinColor`/`FatColor`/`CarbsColor`/`CalorieColor`). The kcal↔macro consistency-warning `Card` is flat (`surface` fill, 0 elevation, 1dp `error` hairline border, `error`-tinted icon + title). Full-width accent "Speichern" button (14dp radius). `MacroCalculator` recommendations, the warning condition, and target-weight handling are unchanged.
- **Local dates.** `FoodEntry.date` and `WeightEntry.date` are `LocalDate` in the domain; stored as ISO-8601 `String` in Room.
- **DailyGoal** is a singleton row (`id = 1`, `OnConflictStrategy.REPLACE`) — no per-day goal history. Holds the macro/kcal goals plus an optional `targetWeightKg` (nullable; set in the Settings goal editor, shown as a target line in the weight stats).
- **WeightEntry** is one per day, enforced via a `UNIQUE` index on the `date` column + `OnConflictStrategy.REPLACE` on insert.
- **Day navigation** in `OverviewViewModel` is driven by a `MutableStateFlow<LocalDate>`. The `uiState` uses `flatMapLatest` to re-subscribe to food entries and today's weight whenever the date changes.
- **AddFoodScreen lists**: the top bar holds a rounded `surface`-filled search field (14dp radius, hairline border, `Icons.Rounded.Search` leading icon) next to a **separate 56dp square barcode tile** (hairline-outlined, `Icons.Rounded.QrCodeScanner`) — no longer an inline trailing icon. When the search field is empty, a **custom mono-uppercase tab row** (`MonoTab`, 2dp accent underline on the active tab — replaces Material `TabRow`) switches between two lists — "Verlauf" (tab 0, left, **shown by default**: last 500 entries grouped by date, mono-uppercase `HEUTE`/`GESTERN`/… group headers via `DateGroupHeader`) and "Meine Lebensmittel" (tab 1, right: custom foods). When a query is typed, a flat filtered list replaces both sections: matching custom foods first, then deduplicated history entries (one per food name, most recent). List rows share one `FoodRowContent` composable (tag dot + name/brand in sans + mono detail line). Tapping any item opens the **amount bottom sheet** (add to log). Custom foods support swipe gestures: StartToEnd (left-to-right) opens the edit dialog; EndToStart (right-to-left) triggers deferred delete with undo snackbar. History entries support EndToStart swipe to delete (also with undo).
  - **Amount entry — `AmountSheet`** (`ModalBottomSheet`, 28dp top radius, scrim; replaced the former `AlertDialog`): food name (serif `titleLarge`), a `MENGE` mono caption over a serif-valued `NumericTextField` (`label = null`, `g` suffix), a `MAHLZEIT` mono caption + horizontally-scrollable `MealCategory` `FilterChip`s, `TagSelector` (its own `TAG` caption), a `surface-2` pill macro preview (mono, shown when amount > 0), and a full-width accent "Hinzufügen" `Button` (enabled only when amount > 0). All field captions are mono-uppercase `text-2` (matching the Goals editor). Same flow/fields as before (select-all-on-focus via `NumericTextField`, tap-selected-tag-chip clears to `NONE`); dismiss via scrim/swipe → `dismissSelection`.
  - **Bottom sheets everywhere**: the same `ModalBottomSheet` pattern (28dp top corners, `surface` container, `navigationBarsPadding()` + `imePadding()`, full-width accent confirm button) backs all the app's data-entry surfaces — the AddFood amount entry (`AmountSheet`), the custom-food editor (`CustomFoodDialog`, a scrollable form sheet), the Overview edit flow (`EditFoodDialog`), and the barcode manual number entry. Only the barcode-error popup and the weight-entry prompt (`WeightCard`) remain `AlertDialog`s (small single-purpose confirmations).
- **Meal sections** (OverviewScreen): entries are grouped by `MealCategory` (order: breakfast, lunch, dinner, snack). Each non-empty section shows a header with the meal name and its total kcal (e.g. "Frühstück — 729 kcal", `kcalForMeal` on the UI state). For a **main meal** (`MealCategory.isMainMeal`, i.e. not snack) that is empty on the viewed day but has entries on the previous day, a "<Meal> von Gestern kopieren" button is shown instead of the section. `OverviewUiState.copyableMeals` computes these; `OverviewViewModel.copyMealFromPreviousDay(category)` re-inserts the previous day's entries of that meal into the viewed day (new ids/timestamps, same foods/amounts/category). The ViewModel subscribes to both the viewed day's and the previous day's entries via `combine`. Snacks are never copyable.
- **Swipe gestures on food entries** (OverviewScreen): EndToStart (right-to-left) shows a red delete background and deletes the entry on full swipe. StartToEnd (left-to-right) shows a green edit background and opens the edit bottom sheet (swipe snaps back via `confirmValueChange` returning `false`). `FoodEntryRow` has an explicit `surface` background so the swipe background only shows when actively swiping. There is no separate edit button — swiping is the only way to edit or delete.
- **Clean-eating tags** (`domain/model/FoodTag`): every food carries a `FoodTag` — `HEALTHY` (green), `NEUTRAL` (amber), `UNHEALTHY` (red), or the default `NONE` (grey/untagged). Only `HEALTHY` counts as "clean" (`FoodTag.isClean`); untagged and non-healthy kcal count against the clean share, so the ~80–90% target rewards tagging. The tag is stored per `FoodEntry` **and** per custom `Food`, so it only has to be set once and is then pre-filled whenever the food is reused (from history or custom foods) — non-retroactive (editing the tag on one entry does not rewrite past entries).
  - **Where set**: a shared `TagSelector` chip row (`ui/components/FoodTagUi.kt`, mono-uppercase `TAG` caption) in the AddFood amount bottom sheet, the Overview edit sheet, and the custom-food editor. Tapping the selected chip clears back to `NONE`. `TagDot` shows a small colour dot before the food name in every list. Tag colours live in `ui/theme/Color.kt` (`TagHealthyColor`/`TagNeutralColor`/`TagUnhealthyColor`); `FoodTag.color()`/`displayName()` are UI-layer extensions in `FoodTagUi.kt`.
  - **Segmented calorie ring** (`OverviewScreen.CalorieRing`): a 176dp ring (stroke ~15.5dp, track = `surfaceVariant`/surface-2) drawn as consumed-kcal arcs split by tag (green → amber → red → grey/untagged), each sized by its share of consumed kcal within the filled `current/goal` portion (butt caps so segments abut). Centre: kcal in serif `displayLarge`, `KCAL` mono micro-label, and a sans caption ("noch X" / "+X zuviel" / "Ziel erreicht"). `OverviewUiState.kcalForTag`/`cleanKcal`/`cleanPercent` back it; a `CleanEatingSummary` ("N % CLEAN" in mono) + tag legend show under the ring when any tag is present. No target is shown — the caption reports the clean share only.
- **Overview visual language** ("Ink & Paper"): the summary + weight cards are flat (`surface` fill, 1dp `outlineVariant` hairline border, zero elevation, 24/18dp radii); macro bars use a sans-medium label, a mono value, and an 8dp pill track tinted 20 % of the bar colour; secondary macros show a serif value with a small sans unit and a `text-3` (`outline`) label; meal-section headers use a serif accent name + mono `text-2` kcal; the copy-from-yesterday control is a hairline-outlined pill (`OutlinedButton`, `CircleShape`); the FAB is a 20dp-radius accent squircle with dark `onPrimary` icon and no elevation. The bottom nav (`AppNavigation`) is flat (`surface`, `tonalElevation = 0`, hairline top `HorizontalDivider`), active = accent + filled (Rounded) icon, inactive = `text-3` + `Icons.Outlined.*` variant.
  - **Entrance/replay motion** (`OverviewScreen`): the calorie ring and macro bars animate in from 0 on (a) screen open, (b) day change, and (c) a tap on the summary card. Driven by a single master `Animatable` "clock" (seconds) restarted via `LaunchedEffect(state.date, replayTrigger)` — `replayTrigger` is a UI-local `Int` bumped on the card tap (no ViewModel/domain state; the card is `clickable` with `indication = null`). Each element derives its own eased sub-progress from a fixed window on the clock via `windowProgress(time, start, end, easing)`: ring segments draw sequentially in kcal order across ~0.5–2.05s (per-segment window proportional to its kcal share, `EaseOutCubic`), the centre kcal counts up 0.4–2.0s (`EaseOutExpo`), the sub-caption fades in 2.1–2.6s, the clean summary fades+slides in 2.6–3.2s; macro bars grow + count up staggered (Protein 0.4–1.3s / Kohlenhydrate 0.55–1.45s / Fett 0.7–1.6s, `EaseOutCubic`). Timings mirror `design_handoff_macrotrac_redesign/Calorie Ring Motion.dc.html` / `Macro Bars Motion.dc.html`.
  - **Clean trend** (`StatsScreen`): a "Clean-Ernährung" card charts the clean share per bucket (`StatsUiState.cleanPoints`, 0–100 % fixed scale, via `CleanBarChart` — no target line) with a range-wide average header (`overallCleanPercent` = total healthy kcal / total kcal). `StatsViewModel.bucketedPoints` is a shared aggregator used by both the calorie and clean charts so they stay on the same time axis.
- **Stats screen** (`ui/stats/`) uses Canvas-based charts (no chart library dependency). Calorie data is aggregated daily for 7-day/30-day views and monthly for the 1-year view. The time range drives a `flatMapLatest` to re-subscribe to both food and weight repositories. It is a **pure insight surface** — the "Daten" section (backup + reminder toggle) moved to the Settings hub, so `StatsScreen` hosts only `StatsViewModel`. A trailing `Spacer` gives the last card breathing room above the bottom nav.
  - **Visual language** ("Ink & Paper"): the three chart cards are flat (`surface` fill, 1dp `outlineVariant` hairline border, zero elevation, 20dp radius) like the Overview cards. The time-range selector is a row of mono-uppercase pill `FilterChip`s (`RoundedCornerShape(50)`, active = accent `primary` fill + `onPrimary`/ink label). Calorie bars are `cal`-colored with **rounded tops** (`drawRoundRect` extended past the baseline so the Canvas clips the bottom corners) over a `surfaceVariant` track + dashed accent goal line; the clean chart is unchanged (`health` bars, square). The weight `WeightSummary` shows mono-uppercase (`labelSmall`) labels over serif (`titleMedium`) values; the weight line colors are raw = `protein`, trend = `tertiary` (fat tone), target = accent, gridlines = `surfaceVariant`. Export/Import buttons use 14dp rounded-rect shape.
  - **Weight chart**: the weight-series math lives in the pure, Android-free `WeightSeries` object (unit-tested in `WeightSeriesTest`); the ViewModel wraps its output in `WeightChartData`. Key properties: (1) **time-proportional x-axis** — points are placed by their real `LocalDate` within `[rangeStart, rangeEnd]`, not by index, so date gaps render truthfully; (2) **kg y-axis** with min/mid/max gridline labels (padded, rounded to half-kg via `WeightSeries.bounds`) plus a `WeightSummary` header (current / change / target); (3) **YEAR view aggregates** one averaged sample per calendar month (placed mid-month); (4) **moving-average trend** overlay (`WeightSeries.movingAverage`, trailing window per range: none for WEEK, 7d MONTH, 90d YEAR — raw line dims to make room for it) and a **dashed target line** from `DailyGoal.targetWeightKg`. Canvas text (axis labels) is drawn via `nativeCanvas`/`android.graphics.Paint`.

  - Text search endpoint is no longer used; search is local-only.

- **Daily meal reminder** (`notification/`): if no food entry exists for today by **17:00**, an unobtrusive notification is posted (channel importance `LOW` → no sound/heads-up). Weight entries do **not** count — only meals.
  - `MealReminderScheduler` enqueues a **self-rescheduling one-time** `WorkManager` job (`ExistingWorkPolicy.REPLACE`) whose initial delay targets the next 17:00; `MealReminderWorker` re-enqueues the next day's run in a `finally` block. `MacroTracApp.onCreate` also calls `schedule()` so the job stays aligned to the next 17:00. The next-17:00 math is the pure, unit-tested `initialDelayMillis(now)`.
  - The worker gets its repositories via a Hilt `@EntryPoint` (`EntryPointAccessors.fromApplication`) — deliberately **not** `@HiltWorker`, so no custom `WorkerFactory`/`Configuration.Provider` is needed and the default WorkManager initializer stays in use.
  - `MealReminderNotifier` builds the notification and taps open `MainActivity`. `POST_NOTIFICATIONS` (Android 13+) is declared in the manifest and requested in `MainActivity.onCreate`.
  - **Enable/disable**: `SettingsRepository` (domain interface, `SettingsRepositoryImpl` backed by `SharedPreferences` "macrotrac_settings") stores `meal_reminder_enabled` (default **true**). The toggle lives in the **Settings hub "Daten" section** ("Tägliche Erinnerung"), a flat card with a pill-styled `Switch` (accent `primary` track + `onPrimary` thumb when on; `surfaceVariant`/surface-2 track + `outline` thumb when off, transparent borders). The worker reads the flag each run and no-ops when disabled (still reschedules), so toggling needs no scheduler coupling.
- **Workout module** (data model + persistence + exercise catalog & UI — see `docs/workout-spec.md`). Lives parallel to the nutrition side and only touches it at defined points (later: body weight, date). Domain models are Android-free: `Exercise` (catalog + custom), `WorkoutTemplate`/`TemplateExercise`, `WorkoutSession`/`SessionExercise`/`SetEntry`, plus enums `ExerciseType` (WEIGHT_REPS/BODYWEIGHT — additively extensible), `SetType` (WARMUP/NORMAL/DROP/FAILURE), `Mechanic` (COMPOUND/ISOLATION, nullable). Each enum has a defensive `parse` (fallback to a default / null).
  - **Stable string keys.** Everything references an exercise by its `stableId` (catalog = free-exercise-db id, custom = generated UUID), **never** the Room row id — keeps cross-device backup import (a later step) consistent. `WorkoutTemplate`/`WorkoutSession` also carry a `stableId` (UUID) for the same reason. Within the DB, parent→child links (template→its exercises, session→exercises→sets) use row-id foreign keys with `onDelete = CASCADE`.
  - **Session status.** `WorkoutSession.isActive` marks the single in-progress session (persisted continuously so it survives app death / can be resumed); `activeSession()` is the indexed lookup. At most one active session is an app-logic invariant (SQLite can't express a partial unique index Room would validate).
  - **Room shape.** Entities under `data/local/entity/`, nested read POJOs under `data/local/relation/` (`TemplateWithExercises`, `SessionWithExercises` → `SessionExerciseWithSets`, two-level `@Relation`). DAOs are interfaces; `@Transaction @Query` for graph reads. `Exercise` list fields (muscles, instructions) are stored newline-joined in one column and split in mapping (no Room `TypeConverter` introduced). Ordering isn't guaranteed by `@Relation`, so children carry a `position` and mapping sorts by it; on save, positions are re-numbered from list order (canonical).
  - **Repositories.** `ExerciseCatalogRepositoryImpl` (upsert-by-`stableId` via `@Insert(REPLACE)` on the unique index — row-id churn is harmless since nothing references it), `WorkoutTemplateRepositoryImpl`, `WorkoutSessionRepositoryImpl`. Composite writes (a template/session + its children) run inside a single transaction via an injected **`TransactionRunner`** (`RoomTransactionRunner` wraps `db.withTransaction`; a pass-through fake makes the repos unit-testable without Room). Mapping is the pure `WorkoutMappers`.
  - **Exercise catalog snapshot (`data/seed/`).** The bundled catalog is the `free-exercise-db` (yuhonas, **Public Domain**) dataset, shipped **data-only, no images** as a single minified asset `app/src/main/assets/exercise_catalog.json` (~770 KB, 873 exercises; a JSON array with only the fields we use: `id`/`name`/`primaryMuscles`/`secondaryMuscles`/`equipment`/`mechanic`/`category`/`instructions`). `ExerciseSnapshotParser` (pure Moshi, Android-free) parses it into `Exercise`s, deriving `type` from equipment (`"body only"` → `BODYWEIGHT`, else `WEIGHT_REPS`) and marking them non-custom. **No runtime API** — access is only ever through the `ExerciseCatalog` repository interface, so the source is swappable without touching feature code.
  - **Seeding.** `ExerciseCatalogSeeder.seedIfNeeded()` (Android-free; injected `ExerciseCatalogRepository` + `SeedVersionStore` + `ExerciseSnapshotSource`) seeds once per snapshot version: it no-ops when the stored version ≥ `SNAPSHOT_VERSION` (constant — bump it when the asset changes), else `upsertAll`s the parsed snapshot and records the version. Idempotent and custom-preserving because the upsert keys on `stableId` (catalog ids vs. custom UUIDs never collide). Wired in DI as `SharedPrefsSeedVersionStore` (own `macrotrac_catalog` prefs) + `AssetExerciseSnapshotSource` (reads the asset off the main thread); `MacroTracApp.onCreate` launches `seedIfNeeded()` on a background scope.
  - **Training tab UI** (`ui/workout/WorkoutScreen` + `WorkoutViewModel`): a searchable/filterable exercise list in the Ink & Paper style, reusing the AddFood building blocks (rounded `surface` search field, horizontally-scrollable `FilterChip` rows for muscle + equipment with mono `MUSKEL`/`EQUIPMENT` captions, a `FoodRowContent`-style `ExerciseRowContent` = name + mono detail line, `ModalBottomSheet`s). Filters/search live on `WorkoutUiState`; the filtered list and `filterOptions` (distinct sorted muscles/equipment) are `WhileSubscribed` StateFlows on the catalog. Catalog exercises are **read-only** (a `Lock` glyph on the row); tapping any row opens a read-only detail sheet (type/muscles/equipment/mechanic/category/instructions). Custom exercises are created via the accent FAB → `ExerciseEditorSheet` (name, `ExerciseType` chips, comma-separated muscles, equipment, `Mechanic` chips, one-line-per-step instructions), and edited/deleted via swipe (StartToEnd = edit, EndToStart = deferred delete with undo snackbar) — same pattern as custom foods. The VM mints a UUID `stableId` for new customs and `upsertAll`s a single-element list; delete is by `stableId`.
