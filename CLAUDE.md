# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Development workflow

Every feature or change follows this workflow — keep it in mind from the first line of code:

**When in doubt, ask.** Any uncertainty about *what* to build — the requirement, the desired behaviour, the functional scope — must be resolved by asking the user before implementing. Never guess at intent. *Technical* decisions (architecture, patterns, data structures, naming, test design, refactors), by contrast, are made autonomously to the highest programming standards — do not ask about those, just apply best practice. In short: ask about the functional/feature side, decide the engineering side yourself.

1. **Plan deliberately.** Design so the software stays *extensible* (new cases slot in without rewrites) and *well testable* (logic separable from Android/UI, behaviour reachable by a test). Prefer a reusable building block over a one-off when the same need will recur.
2. **Implement** the plan.
3. **Remove dead code.** Delete now-unused sections, wrappers, imports, and helpers so the codebase stays a clear picture of what's actually in use — no leftovers from the previous shape of the code.
4. **Validate** the change against the requirement (and edge cases) by reading it back.
5. **Write tests** covering the new behaviour (ViewModel logic → JVM unit tests under `app/src/test/`; UI behaviour → Compose tests under `app/src/androidTest/`).
6. **Compile and test from the command line** using the commands below — `assembleDebug` + `testDebugUnitTest` (and `connectedDebugAndroidTest` for instrumented tests when a device/emulator is attached). Don't consider a step done until it builds and tests pass.
7. **Update the docs** — keep this `CLAUDE.md` in sync with new architecture, design decisions, and conventions.
8. **Commit** with a clear, conventional message.

### Tests

Unit tests live in `app/src/test/` and run on the JVM (`testDebugUnitTest` — no Robolectric). Conventions:

- **Fakes, no mocking library** (`.../fake/`): `FakeFoodEntryRepository`, `FakeGoalRepository`, `FakeFoodSearchRepository`, `FakeWeightRepository`, `FakeCustomFoodRepository`, `FakeSettingsRepository`, and for the workout module `FakeExerciseCatalogRepository`, `FakeWorkoutTemplateRepository`, `FakeWorkoutSessionRepository` (the session fake encodes the "at most one active session" invariant).
- Tests use `kotlinx-coroutines-test` + `turbine`. **Every ViewModel has a test class** covering its full behaviour; keep that invariant when adding VM logic. A `WhileSubscribed` StateFlow only computes real values while collected — tests keep it hot with a `backgroundScope` collector and then read `.value`.
- **Pure domain/series logic gets its own test class** (`MacroCalculatorTest`, `InlineHistoryTest`, `RestTimerTest`, `WorkoutMetricsTest`, `TemplateUpdateTest`, `ExerciseHistoryTest`, `WeightSeriesTest`, `WorkoutSeriesTest`, `WorkoutFormatTest`, …) — cover the edge cases, not just the happy path.
- Entity↔domain mapping is kept in pure, Android-free helpers so it can be unit-tested without Room (`WorkoutMappers` → `WorkoutMappersTest`; the CSV formats → `BackupImporterTest`/`WorkoutBackupTest`). Repository behaviour is tested with in-memory fake DAOs + a pass-through `TransactionRunner` (`WorkoutRepositoriesTest`).
- **Locale-sensitive formatting in tests**: `currentAppLocale()` falls back to `Locale.getDefault()` on the JVM, so a test asserting formatted decimals must pin `Locale.setDefault()` in setup and restore it afterward (see `WorkoutFormatTest`) — otherwise the expected separator depends on the machine running the test.
- `app/src/androidTest/` holds the **instrumented** tests (run with `connectedDebugAndroidTest` against an emulator/device): `WorkoutMigrationTest` exercises the real Room migrations v7→v8→v9→v10→v11 on Android's SQLite (creates the old DB, runs the migration, asserts new tables/columns + preserved data + FK cascade). Room's schema JSON is exported to `app/schemas/` (`exportSchema = true` + the `room.schemaLocation` KSP arg); a migration's CREATE statements are copied verbatim from the generated schema JSON so the migrated schema matches the entities exactly.

### Backup & data portability

- **Auto Backup**: `backup_rules.xml` (pre-12) and `data_extraction_rules.xml` (Android 12+) explicitly include `trainist.db` + WAL files — Android backs these up to Google Drive automatically, no code required.
- **Full Backup Export** (`data/backup/BackupExporter`): reads all data — nutrition (food entries, weight entries, daily goal, custom foods) and training (custom exercises, templates + slots, sessions + exercises + sets) — and writes a ZIP to `cacheDir` with one named CSV per section (`food_entries.csv`, `weight_entries.csv`, `daily_goal.csv`, `custom_foods.csv`, `exercises.csv`, `workout_templates.csv`, `template_exercises.csv`, `workout_sessions.csv`, `session_exercises.csv`, `set_entries.csv`), shared via `FileProvider` + `Intent.ACTION_SEND`. Only **custom** exercises are exported — catalog entries are reproduced by the bundled-snapshot seeding on every install and resolve by stable id.
- **Full Backup Import** (`data/backup/BackupImporter`): opens a user-picked URI, auto-detects ZIP vs single CSV via the PK magic bytes. A ZIP imports every known section by entry name (unknown/missing entries ignored); a single CSV is routed by `detectCsvType` (column presence — see below). The Android IO is kept thin; dispatch/parse/save lives in the Android-free `importZipEntries(InputStream, ImportTargets)` / `importCsvLines(List<String>, ImportTargets)`, unit-tested end-to-end in `BackupImporterTest` (incl. legacy nutrition-only backups, a full training round-trip and a fresh-device relink).
- **Relational training data — stable-key reassembly** (`data/backup/WorkoutBackup.kt`, pure, `WorkoutBackupTest`): the training graph is split into flat CSVs whose **foreign keys are the stable string keys** (`Exercise.stableId`, `WorkoutTemplate.stableId`, `WorkoutSession.stableId`), **never** row ids. On import, `assembleTemplates`/`assembleSessions` re-group the rows (`set_entries` link to their parent by `(session_stable_id, exercise_position)`; rows sorted by `position`), then each root is saved fresh (`id = 0`, target mints its own row ids). Custom exercises are `upsertAll`-ed first so references resolve on a fresh device. The relational sections only reassemble as a whole ZIP; a lone `exercises.csv` imports standalone, the other five training CSVs are recognised-but-no-op on their own.
- **CSV type detection** (`detectCsvType`): identifies a CSV by a unique column. Training sections are matched **first** (`set_entries.csv` also carries `weight_kg`, so it must win over the weight-entries rule):
  - `is_custom` → `EXERCISES`; `template_name` → `WORKOUT_TEMPLATES`; `target_sets` → `TEMPLATE_EXERCISES`; `is_active` → `WORKOUT_SESSIONS`; `superset_group_id` → `SESSION_EXERCISES`; `exercise_position` → `SET_ENTRIES`
  - `food_name` → `FOOD_ENTRIES`; `weight_kg` → `WEIGHT_ENTRIES`; `kcal_per_100g` → `CUSTOM_FOODS`; `kcal` (without the above) → `DAILY_GOAL`
- **CSV formats** (all pure Kotlin, one object per section, name-based columns; `CsvFormat.parseHeaders` is the shared header parser): `CsvFormat` (food entries, incl. `tag` → `FoodTag.parse`, missing → `NONE`), `CustomFoodCsvFormat`, `WeightCsvFormat` (`date`, `weight_kg`, `timestamp_ms`), `GoalCsvFormat` (single row; optional `target_weight_kg`), `ExerciseCsvFormat`, `WorkoutTemplateCsvFormat`, `TemplateExerciseCsvFormat` (`set_types` = planned `SetType`s joined by `|`; older backups fall back to `target_sets` NORMAL sets), `WorkoutSessionCsvFormat` (optional `template_stable_id`), `SessionExerciseCsvFormat`, `SetEntryCsvFormat`. List cells (muscles/instructions/set types) are `|`-joined — one line per row; an embedded newline would break the line-based reader.
- **FileProvider** authority: `dev.antonlammers.trainist.fileprovider`, paths in `res/xml/file_paths.xml` (cache dir).

### Backup schema evolution — backward compatibility contract

All CSV parsing is name-based (column order irrelevant). This gives forward and backward compatibility for free, as long as these rules are followed:

| Change | Export (`toRow` / `HEADER`) | Import (`fromRow`) |
|---|---|---|
| **Add a column** | Add constant to the format object, add to `HEADER` and `toRow` | Read with a sensible default for missing values — old exports still parse |
| **Remove a column** | Remove from `HEADER` and `toRow` | Remove the read; unknown columns are ignored, so new exports still parse in old app versions |
| **Rename a column** | Write both old and new names for at least one release, then drop the old | Read the new name with fallback to the old until the transition ends |
| **Add a new data type** | New `*CsvFormat` object + ZIP entry name in `BackupExporter` | New `CsvType` variant + detection rule in `detectCsvType` (more-specific rules before any that shares a column), handle in `importZipEntries` + `importCsvLines` |
| **Add a relational data type** | Flat CSVs whose foreign keys are **stable string keys** (never row ids); one section per table | Reassemble the graph in a pure helper (like `WorkoutBackup`), save each root with `id = 0`; keep it a whole-ZIP operation |
| **Change the ZIP container format** | — | Extend `isZip` / add a new container detection path; keep old paths for legacy files |

**No explicit version field is needed.** Structural detection (column presence, ZIP magic bytes, entry names) is sufficient. Only introduce a version field if two formats become structurally ambiguous.

**Referential integrity across devices.** Relational backups never serialise auto-increment row ids — all cross-row references use stable string keys, so a backup rebuilds correctly on a device whose row ids differ.

### Key design decisions

- **Domain layer is Android-free.** ViewModels depend only on `domain/repository` interfaces. Swapping the data source touches only `data/` and `di/`.
- **Bottom navigation — four tabs** (`AppNavigation`, `bottomNavItems()`): **Ernährung** (`Screen.Overview`), **Training** (`Screen.Workout`, the template list), **Statistik** (`Screen.Stats`), **Einstellungen** (`Screen.Settings`). Icons active = `Icons.Rounded.*`, inactive = `Icons.Outlined.*` (Restaurant / FitnessCenter / BarChart / Settings). The nav is **icon-only** — no `label`; the localized name lives on as each icon's `contentDescription`. Goal editing lives in the Settings hub (no Goals tab). Any route not in `bottomNavItems` (`AddFood`, `BarcodeScanner`, `ExerciseCatalog`, `ExerciseDetail`, `TemplateEditor`, `WorkoutSession`, `WorkoutHistory`) hides the bar.
- **Theme isolation — "Ink & Paper".** Replacing `ui/theme/Color.kt`, `Theme.kt`, and `Type.kt` restyles the entire app. Fixed monochrome color system (no Material You, `dynamicColor = false`): dark = "Ink", light = "Paper", chosen from the system setting (`isSystemInDarkTheme()`), no in-app toggle. Colour is reserved for data. `Color.kt` holds the `Ink*`/`Paper*` palettes (paper/bg/surface/surface2/line/text/text2/text3), the mode-independent `AccentLilac` (`#B79FCB`, `primary`/`secondary`) + `OnAccentInk`, the pastel macro colours (`CalorieColor`/`ProteinColor`/`CarbsColor`/`FatColor`) and the clean-eating tag colours. `Theme.kt` maps these onto the Material3 schemes (`surfaceContainerLowest` = chrome, `outlineVariant` = hairline, `error` = unhealthy tone). The base XML theme `Theme.Trainist` must stay **DayNight-aware** (`values/themes.xml` light + `values-night/themes.xml` dark window background) — otherwise the window background bleeds through as a hairline at the bottom-nav seam in dark mode.
- **Typography — three downloadable Google fonts** (`ui/theme/Type.kt`): `NewsreaderSerif` (display/headline/title: screen titles, big numbers), `HankenGroteskSans` (body + `labelLarge`), `JetBrainsMono` (`labelMedium`/`labelSmall` uppercase micro-labels with wide tracking, e.g. `KCAL`). Loaded at runtime via the Google Fonts provider (`font_certs.xml`) — no bundled font binaries; requires Google Play Services at runtime.
- **Icons — Material Symbols Rounded.** All icons use `Icons.Rounded.*` (not Filled/Default). Reach for the Rounded variant when adding a new icon.
- **Numeric input — `ui/components/NumericTextField`.** *Every* numeric field uses it, never a bare `OutlinedTextField`. On focus it **selects its whole content** so a pre-filled value is overwritten by simply typing. String-based `value`/`onValueChange` API (callers parse, accepting comma or period via `normalizeDecimal`); `decimal = false` for whole-number keyboards; nullable `label`; optional `textStyle`/`leadingIcon`/`placeholder` slots (the placeholder shows grey inline-history hints in the session).
- **Drag-to-reorder — `ui/components/DragReorderColumn`.** The one reordering pattern app-wide (template editor slots, session sets, templates list, stats cards): long-press the row's drag handle (`Icons.Rounded.DragIndicator`, haptic tick), drag, settles on release. Renders a plain non-lazy `Column` (nest it as a single item inside an outer `LazyColumn`); takes `items`/`key`/`onMove(from, to)` plus an `itemContent` slot receiving a `rowModifier` (apply to the row container — carries drag translation/elevation) and a `dragHandleModifier` (apply only to the handle so the gesture never fights the row's own controls). While dragging it calls `onMove` **once per adjacent step** (whenever the dragged row crosses a neighbour's mid-height), so the ViewModel side is a simple adjacent swap persisted continuously — no "commit on drop" step. Internally each row **must be wrapped in `key(itemKey) { … }`** — without it the non-lazy `Column` identifies children by slot, so the first swap restarts the row's `pointerInput` and cancels the in-progress gesture. `key` must be genuinely unique per row (duplicates of the same exercise are allowed), so rows carry stable ids (`TemplateSlot` a synthetic negative `newId()`; session/set entries already have stable ids).
- **Settings hub** (`ui/settings/SettingsScreen`): goal editor + "Daten" (backup export/import + reminder toggle) + "Sprache" (language picker) + "Unterstützung" (donation/contact) in one scrollable screen; co-hosts `GoalsViewModel`, `DataViewModel`, `LanguageViewModel`. Top-level tab, no back button — "Speichern" shows a snackbar. Section composables are `ColumnScope` extensions so `Modifier.align` works. The donation section is static UI (PayPal link + `mailto:` intent to `DEVELOPER_CONTACT_EMAIL`) — no ViewModel. Goal fields (`FieldLabel`/`GoalField`/`AccentTick`/`formatWeight`) live `internal` in `ui/goals/GoalFields.kt` because the onboarding goals guide reuses the exact same look.
- **First-launch onboarding** (`ui/onboarding/`): welcome screen with three ways in — **Schnellstart** (backup import via `DataViewModel.import` with an `onSuccess` callback), **Einfach loslegen** (empty start), **Ziele einrichten** (goals guide sharing `GoalsViewModel`/`GoalFields`; the designated home for future setup steps). Gated by `SettingsRepository.isOnboardingCompleted()` (SharedPreferences, default false). `OnboardingViewModel` reads the flag once into `Loading`/`Onboarding`/`Completed`; `MainActivity.setContent` switches on it (`Loading` renders nothing to avoid a flash). Every path calls `complete()`, which flips state synchronously and persists.
- **Local dates.** `FoodEntry.date` and `WeightEntry.date` are `LocalDate` in the domain; stored as ISO-8601 `String` in Room.
- **DailyGoal** is a singleton row (`id = 1`, REPLACE) — no per-day goal history. Holds macro/kcal goals + optional `targetWeightKg` (target line in the weight stats).
- **WeightEntry** is one per day (`UNIQUE` index on `date` + REPLACE on insert).
- **Day navigation** in `OverviewViewModel` is a `MutableStateFlow<LocalDate>`; `uiState` uses `flatMapLatest` to re-subscribe whenever the date changes.
- **AddFoodScreen**: search field + separate square barcode tile in the top bar. Empty query → a custom mono-uppercase tab row (`MonoTab`) with "Verlauf" (default: last 500 entries, grouped by date) and "Meine Lebensmittel" (custom foods); a typed query → one flat filtered list (custom foods first, then deduplicated history). Rows share `FoodRowContent`. Tapping opens the **amount bottom sheet** (`AmountSheet`: amount `NumericTextField`, `MealCategory` chips, `TagSelector`, macro preview, accent confirm). Custom foods: StartToEnd swipe = edit, EndToStart = deferred delete with undo; history entries: EndToStart delete with undo. Search is **local-only** (history + custom foods) — Open Food Facts is used solely for barcode lookup.
- **Bottom sheets everywhere**: one `ModalBottomSheet` pattern (28dp top corners, `surface` container, `navigationBarsPadding()` + `imePadding()`, full-width accent confirm) backs all data-entry surfaces (amount sheet, custom-food editor, Overview edit sheet, barcode manual entry, exercise picker/editor, language picker). Only small single-purpose confirmations stay `AlertDialog`s. **Every `ModalBottomSheet` must pass `modifier = Modifier.statusBarsPadding()`** — Material3's default insets only reserve space at the bottom, so without it a tall sheet draws under the status bar; putting it on the sheet's own `modifier` (not the content) shrinks the height constraint before the sheet picks its expanded offset.
- **Meal sections** (OverviewScreen): entries grouped by `MealCategory` (breakfast, lunch, dinner, snack), each header showing the meal's kcal total. An empty **main meal** with entries on the previous day shows a "<Meal> von Gestern kopieren" button instead (`OverviewUiState.copyableMeals`; `copyMealFromPreviousDay` re-inserts with new ids/timestamps). The VM subscribes to both days via `combine`. Snacks are never copyable.
- **Swipe gestures on food entries** (OverviewScreen): EndToStart = delete (red background), StartToEnd = edit sheet (green background, snaps back via `confirmValueChange` → false). Rows have an explicit `surface` background so the swipe colour only shows while swiping. Swiping is the only edit/delete path — no buttons.
- **Clean-eating tags** (`domain/model/FoodTag`): `HEALTHY`/`NEUTRAL`/`UNHEALTHY`/`NONE`, contributing `cleanWeight` shares of their kcal to the clean total (1.0 / 0.5 / 0.0 / 0.0). Stored per `FoodEntry` **and** per custom `Food` so the tag is set once and pre-filled on reuse — non-retroactive. Set via the shared `TagSelector` (`ui/components/FoodTagUi.kt`, which also holds `TagDot` and the `color()`/`displayName()` extensions). The Overview's segmented **calorie ring** draws consumed kcal split by tag (green → amber → red → grey), with a "N % CLEAN" summary + legend when any tag is present.
- **Overview motion**: the ring and macro bars animate in on screen open, day change, and a tap on the summary card — driven by one master `Animatable` clock restarted via `LaunchedEffect(state.date, replayTrigger)`; each element derives an eased sub-progress from a fixed window on the clock (`windowProgress`). UI-local state only, no ViewModel involvement.
- **Stats screen** (`ui/stats/`): Canvas-based charts, no chart library. Five cards — Kalorien, Clean-Ernährung, Gewicht, Trainingsfrequenz, Kraftverlauf — all rendered through a shared `ChartCard`/`ChartEmptyHint` pair and **user-reorderable** via `DragReorderColumn`; `StatsViewModel.moveCard` persists the order via `SettingsRepository.setStatsCardOrder` (recombined onto the data flow via a *separate* `combine`, so reordering never re-triggers the repository re-subscription driven by `_timeRange`/`_selectedExerciseId`). One shared **`BarChart`** backs every bar chart (dynamic scale or `fixedMaxValue` — the clean chart pins 100 so the axis reads as %; `goalValue > 0` draws a dashed goal line). Its rounded bar tops overdraw past the baseline, so the draw is wrapped in an explicit `clipRect` — `DrawScope` does **not** clip to its own bounds. Time-range selector = mono-uppercase pill `FilterChips`.
  - **Weight chart**: pure series math in `WeightSeries` (`WeightSeriesTest`). Time-proportional x-axis (points placed by real date, gaps render truthfully); kg y-axis with min/mid/max gridlines (`WeightSeries.bounds`); YEAR aggregates one averaged sample per month; moving-average trend overlay (7d MONTH / 90d YEAR) + dashed target line from `DailyGoal.targetWeightKg`. Axis text via `nativeCanvas`.
  - **Training charts**: pure series math in `WorkoutSeries` (`WorkoutSeriesTest`), bucketing over the same `TimeRange` axis. **Trainingsfrequenz** = completed sessions per bucket. **Kraftverlauf** = per-exercise best estimated 1RM per bucket (`strengthSamples`), drawn by `StrengthChart` (`ui/stats/StrengthChart.kt`, `internal`, caller-supplied `tickDates`/`tickFormatter` — also reused by the exercise-detail all-time mini-chart). A scrollable `ExerciseSelector` picks the exercise (auto-defaulting to the first); body weight for bodyweight 1RMs is resolved from **all** weigh-ins so it can reference one before the range start.
- **Open Food Facts** base URL: `https://world.openfoodfacts.org/`, used **only** for barcode lookup (`api/v2/product/{barcode}`, `status == 1` = found). OkHttp's `HttpLoggingInterceptor` is added in **debug builds only** (`BuildConfig.DEBUG` check in `NetworkModule`).
- **Barcode scanner** (`BarcodeScannerScreen`): CameraX + ML Kit. The detected barcode goes back via `NavBackStackEntry.savedStateHandle["barcode"]`; `AddFoodViewModel.handleBarcode()` resolves it. `BarcodeAnalyzer` fires exactly once per scan session (`AtomicBoolean`); the always-docked manual-entry panel funnels through the same `submitBarcode` lambda/guard. Torch toggle only renders when `cameraInfo.hasFlashUnit()`. `ScanOverlay` = darkened canvas with corner brackets + animated scan line.
  - **IME handling — bottom-anchored input over a full-screen edge-to-edge surface**: the screen pins its own soft-input mode to `SOFT_INPUT_ADJUST_NOTHING` for its lifetime (a `DisposableEffect` on the activity window, restoring the previous mode `onDispose`), then uses `Modifier.navigationBarsPadding().imePadding()` on the panel. Without the pin, the window resize *and* the reported IME inset double up and the panel floats a keyboard-height too high. This is the general fix pattern for this situation — don't fight the resize with padding math.
- **`@ExperimentalGetImage`** propagates from `BarcodeAnalyzer` → `BarcodeScannerScreen` → `AppNavigation` → `MainActivity`. Expected, not a warning to suppress.
- **Daily meal reminder** (`notification/MealReminder*`): if no food entry exists for today by **17:00**, a low-importance notification is posted (weight entries don't count). `MealReminderScheduler` enqueues a **self-rescheduling one-time** WorkManager job (`REPLACE`; the worker re-enqueues in `finally`; `TrainistApp.onCreate` also schedules). The next-17:00 math is the pure, tested `initialDelayMillis(now)`. The worker gets its repositories via a Hilt `@EntryPoint` — deliberately **not** `@HiltWorker`, so the default WorkManager initializer stays in use. Toggle: `meal_reminder_enabled` in `SettingsRepository` (default true); the worker reads the flag each run and no-ops when disabled (still reschedules), so toggling needs no scheduler coupling. `POST_NOTIFICATIONS` (Android 13+) is declared in the manifest and requested in `MainActivity.onCreate`.
- **`SettingsRepository`** (`SharedPreferences` "trainist_settings"): `meal_reminder_enabled`, `stats_card_order` (comma-joined `StatCardType` names; missing/unknown values are patched onto `DEFAULT_ORDER` defensively — the pattern for every persisted enum), `onboarding_completed`, and the app-language wrapper (see i18n).
- **Rest timer** (`domain/RestTimer` + `notification/RestTimer*`): checking off a set starts a rest countdown (per-exercise `Exercise.restSeconds`, else 180 s default). Design constraints that shaped it:
  - **Pure state** (`domain/RestTimer`): anchored to a wall-clock `endAtMs`; remaining time is always recomputed from "now" (correct across backgrounding); `start`/`paused`/`resumed`/`adjusted` are pure transitions.
  - **VM stays Android-free**: `WorkoutSessionViewModel` owns the timer state and emits one-shot `RestCommand`s (`Start`/`Pause`/`Cancel`) on a channel; the screen calls the Context-holding `RestTimerScheduler`/`RestTimerNotifier`. A natural finish clears the in-app bar **without** `Cancel` (the scheduled alert must still fire); pause/skip/finish/discard emit `Cancel`. The in-app `RestTimerBar` (Scaffold `bottomBar`) shows mm:ss + progress with ±15 s / pause-resume / skip; each exercise-card header carries a rest-duration override chip (persisted via `catalog.upsert` with a new `restSeconds`).
  - **Survives leaving the session**: the anchor is persisted onto the session (`restExerciseStableId`/`restTotalSeconds`/`restEndAtMs`/`restPausedRemainingMs`, null when idle) on every transition; on `init` an active session's anchor is restored (dropped + cleared if already expired).
  - **Exact scheduling**: `AlarmManager.setExactAndAllowWhileIdle` (fallback to inexact only if `canScheduleExactAlarms()` is false) — WorkManager/inexact alarms can be deferred by Doze for minutes, which reads as "no alert went off". Uses the auto-granted `USE_EXACT_ALARM` (API 33+) + `SCHEDULE_EXACT_ALARM maxSdkVersion=32`; no runtime request needed.
  - **Ongoing notification** (silent LOW channel `rest_timer_ongoing`): custom content view (`notification_rest_timer.xml`) with a large countdown — running uses an embedded `Chronometer` counting down to the end instant (Android keeps it live without re-posting), paused shows a frozen mm:ss. Text colour is set **explicitly per shade theme** (`uiMode` night mask) — `?android:attr/textColorPrimary` does not reliably resolve to the shade's theme across OEMs.
  - **Expiry alert** (`RestTimerAlertService`, foreground service `mediaPlayback`): the alarm receiver starts the service (exact-alarm firing is a documented FGS-start exemption; only an FGS reliably outlives `onReceive` with the screen locked). It posts a dismissible HIGH-channel heads-up (`rest_timer_alert_v2`) that is itself **silent** — sound/vibration are played explicitly: the system **notification** ringtone (not `TYPE_ALARM` — that's the user's own alarm melody) via `MediaPlayer` with `USAGE_ALARM` audio attributes (audible despite ringer/DND, works over headphones), plus a direct `Vibrator` waveform, holding a partial wake lock. Hard-capped at 6 s (`ALERT_DURATION_MS`) and stoppable via the "Stoppen" action, swipe-away (`setDeleteIntent`), or tapping through to the app — all funnel to `ACTION_STOP`.
  - **Notification-channel rules** (general): a channel's importance/sound/vibration are **immutable once created** — changing any of them requires a **new channel id** (hence the `_v2` suffix on the alert channel). Mutable `name`/`description` self-heal on the next `ensureChannel(s)` call, which runs before every post (this is also what re-localizes channels after a language switch). The alert uses its **own notification id** (2003) distinct from the ongoing one (2002) — Android won't move a posted notification to a different channel on update. `RestTimerNotifier.cancel(context)` dismisses both and stops a still-playing alert service; it runs on pause/skip/finish/discard, tap-to-open, and defensively at the start of every new rest.
  - Tapping any rest-timer notification opens `MainActivity` with `EXTRA_OPEN_WORKOUT_SESSION`; `AppNavigation` reads it and navigates to `Screen.WorkoutSession.start(0)` (resumes the active session regardless of the templateId arg).
- **Workout module** — domain models are Android-free: `Exercise` (catalog + custom), `WorkoutTemplate`/`TemplateExercise`, `WorkoutSession`/`SessionExercise`/`SetEntry`, enums `ExerciseType` (WEIGHT_REPS/BODYWEIGHT — additively extensible), `SetType` (WARMUP/NORMAL/DROP/FAILURE), `Mechanic` (COMPOUND/ISOLATION, nullable). Each enum has a defensive `parse` (fallback to a default / null).
  - **Stable string keys.** Everything references an exercise by `stableId` (catalog = free-exercise-db id, custom = UUID), **never** the Room row id — keeps cross-device backup import consistent. Templates/sessions carry a `stableId` too. Within the DB, parent→child links use row-id FKs with `onDelete = CASCADE`.
  - **Session status.** `WorkoutSession.isActive` marks the single in-progress session (continuously persisted, survives app death, resumable); `activeSession()` is the indexed lookup. At most one active session is an app-logic invariant.
  - **Room shape.** Entities under `data/local/entity/`, nested read POJOs under `data/local/relation/` (two-level `@Relation`). `@Transaction @Query` for graph reads. `Exercise` list fields are newline-joined in one column, split in mapping (no `TypeConverter`). `@Relation` doesn't guarantee order, so children carry `position`, mapping sorts by it, and saves renumber positions from list order (canonical).
  - **Repositories.** Upserts key on `stableId` (`@Insert(REPLACE)` on the unique index — row-id churn is harmless since nothing references it); composite writes run in one transaction via the injected `TransactionRunner` (`RoomTransactionRunner` wraps `db.withTransaction`; a pass-through fake keeps repos unit-testable). Mapping is the pure `WorkoutMappers`.
  - **Exercise catalog snapshot** (`data/seed/`): the `free-exercise-db` dataset (yuhonas, Public Domain), shipped data-only (no images) as `assets/exercise_catalog.json` (~770 KB, 873 exercises). `ExerciseSnapshotParser` (pure Moshi) derives `type` from equipment (`"body only"` → BODYWEIGHT). No runtime API — access only through `ExerciseCatalogRepository`, so the source is swappable. `ExerciseCatalogSeeder.seedIfNeeded()` seeds once per `SNAPSHOT_VERSION` (bump the constant when the asset changes); idempotent and custom-preserving because the upsert keys on `stableId`. `TrainistApp.onCreate` launches it on a background scope.
  - **Training tab structure.** Tab root = the templates list; sub-screens (with back buttons, bottom bar hidden): template editor (`TemplateEditor/{templateId}`, 0 = new), live session (`WorkoutSession?templateId={id}`, 0 = empty), history (top-bar calendar icon), exercise catalog (top-bar library icon), exercise detail.
  - **Templates list** (`TemplatesViewModel`): cards show name + a relative "last used" line, driven by `WorkoutSession.templateStableId` (set once at session creation, null for free workouts; `TemplateListItem` combines templates with the full session history, max date per template). Tap = start a session from the template; StartToEnd swipe = edit; EndToStart = deferred delete with undo; drag handle = manual reorder, persisted immediately via `WorkoutTemplateRepository.reorder` (the templates table's `position` is storage-only — it never appears on the domain model; `save` preserves an existing position via `positionOf`, else appends via `nextPosition`). A resume banner shows above the list while a session is active; the accent Extended FAB starts an empty session.
  - **Live session** (`WorkoutSessionViewModel`): the active session is the source of truth (`MutableStateFlow<WorkoutSession?>`), **persisted on every mutation** (`persist()`, serialized by a `Mutex` so the first insert assigns the row id before follow-ups reuse it — no duplicate rows). On entry the VM resumes an existing active session (template arg ignored); otherwise builds one (from a template: one set per planned `TemplateExercise.setTypes` entry, pre-set to its type) and persists immediately. In-session additions get synthetic negative client ids (`newId()`) as stable UI keys (never persisted — mappers mint real ids). `uiState` joins the session with the catalog (names/types), the session history (inline hints) and the resolved body weight (from `WeightRepository`, for bodyweight-exercise calculations). Set rows: drag handle, `SetTypeBadge` (tap → menu), weight/reps `NumericTextField`s, check-off, delete; cards show a VOLUMEN/E1RM footer once a work set is logged. Set-row text state is local `remember(set.id, set.completed)` so intermediate input survives and adopted values render as real text. Back leaves the session running; `Fertig` finishes (`isActive=false`, `endedAtMs`) — unless total volume is 0, in which case the session is **deleted** instead (empty workouts never clutter the history); `✕` discards after a confirm dialog. A `finished` StateFlow drives `popBackStack`. The `clock: () -> Long` is injected via a secondary constructor (default `System::currentTimeMillis`) so tests use a fixed clock.
  - **Set types**: pickable per set via the badge menu. `ui/workout/SetTypeUi.kt` maps each type to an **existing** colour token (no new colours): NORMAL = `onSurfaceVariant`, WARMUP = `TagNeutralColor`, DROP = `primary`, FAILURE = `error`.
  - **Inline history** (`domain/InlineHistory`): `lastPerformance` returns the ordered set values from the most recent **completed** session containing the exercise (active session ignored); shown as grey placeholders in empty weight/reps fields. **Adopt-on-check-off**: `toggleSetCompleted` adopts the placeholder into a still-empty field (weight == 0.0 / reps == 0) as if typed — entered values are never overwritten; no history → stays empty.
  - **Template sync on finish** (`domain/TemplateUpdate`): finishing a template session offers to fold structural changes back into the template. An exercise is *performed* iff ≥1 set is checked off; its new plan = the completed sets' `SetType`s in order (an added-but-unchecked set does not leak in). Skipped exercises are kept unchanged (never silently shrink a template); live-added performed exercises are appended. `merge` returns null when nothing changed → finish pops straight away; otherwise `pendingTemplateUpdate` drives a "Vorlage aktualisieren?" dialog (confirm saves — same row id, never duplicates). Only the `Fertig` path triggers it — back-navigation and discard never do.
  - **Training calculations** (`domain/WorkoutMetrics`): effective weight = entered weight (WEIGHT_REPS) or tracked body weight + added weight (BODYWEIGHT; `resolveBodyWeightKg` = most recent entry on/before the session date, else earliest known, else null → count only added weight). Volume = Σ effective×reps, warm-ups excluded. Estimated 1RM = Epley (`weight × (1 + reps/30)`), display-only. PR = strictly-greater best effective weight over non-warm-up performed sets (a tie is not a PR); `personalRecords` sweeps the whole history chronologically and flags `(sessionStableId, exerciseStableId)` pairs — a session keeps its badge even once beaten. The PR trophy (`PrBadge`) shows in the history day detail and exercise detail.
  - **History** (`WorkoutHistoryViewModel`): month calendar (Monday-first `weeksOf`) with accent dots on training days; a selected day's sessions render with PR badges, VOLUMEN/E1RM footers, and **editable set rows** (in-place corrections: weight/reps/type/add/remove) + session delete with undo. The day's sessions are held in a local editable copy (`_editable`, reseeded only on day change — `save()` rewrites the set graph and reassigns child ids); calendar markers + PR sweep stay reactive off the repository. `clock` injected for a testable "today".
  - **Template editor** (`TemplateEditorViewModel`): name field + one `SlotCard` per exercise (drag-reorder, remove) + catalog picker sheet (reuses `ExerciseCatalogRepository` on the editor VM — no cross-screen result plumbing, unsaved state never lost). Each slot plans its sets **individually**: one chip per planned set with the shared `SetTypeBadge` (menu per set) + remove, plus "+ Satz" (count clamped 1–20) — a live session started from the template pre-fills each set's type. Save requires a non-blank name + ≥1 exercise; renumbers positions, mints a UUID `stableId` for new templates (preserves on edit). Templates carry no target weight/reps — that's the inline history's job.
  - **Exercise catalog** (`ExerciseCatalogViewModel`): searchable/filterable list (muscle/equipment `FilterChip` rows, clear-✕ in the search field); filtered list + `filterOptions` are `WhileSubscribed` StateFlows. Catalog exercises are read-only (lock glyph); custom exercises via FAB → `ExerciseEditorSheet` (UUID `stableId`, single-element `upsertAll`), edited/deleted via swipe — same pattern as custom foods. Row tap → exercise detail.
  - **Exercise detail** (`ExerciseDetailViewModel`, route `ExerciseDetail/{exerciseStableId}`, reachable from catalog rows and the clickable exercise names in session/history): metadata (`ExerciseFormat`), PR card, all-time strength mini-chart (shared `StrengthChart`), and the full chronological set log. Aggregation is the pure `domain/ExerciseHistory.build` + `WorkoutSeries.strengthHistory`; body weight resolved from all weigh-ins.
- **DB schema**: version 11, exported to `app/schemas/`. Migrations: 1→2 sugar/fiber/mealCategory; 2→3 `weight_entries`; 3→4 `custom_foods`; 4→5 salt columns; 5→6 `targetWeightKg`; 6→7 `tag` on `food_entries` + `custom_foods`; 7→8 the six workout tables (additive); 8→9 the four nullable rest-timer-anchor columns on `workout_sessions`; 9→10 `workout_templates.position` (backfilled to `id` so pre-existing templates keep creation order) + nullable `workout_sessions.templateStableId`; 10→11 nullable `template_exercises.setTypes` (older rows fall back to `targetSets` NORMAL sets).
- **Internationalization.** German is the default (`values/strings.xml` — no `values-de/` folder), English in `values-en/strings.xml`; more languages slot in as additional `values-xx/` folders + entries in `res/xml/locales_config.xml` (referenced via `android:localeConfig`, enables the Android 13+ system per-app language setting). String names follow `<screen_or_module>_<element>_<variant>`, app-wide labels under `common_*` — **grep for the German text before adding a resource**; many labels repeat verbatim across screens. Both files must stay key-identical with matching format specifiers (a `%d`↔`%s` drift between languages is a silent crash — the inactive locale never gets compiler-checked).
  - **Per-app language** via `AppCompatDelegate.setApplicationLocales()`/`getApplicationLocales()` — the single source of truth (delegates to the platform `LocaleManager` on API 33+; persists the choice itself below). `SettingsRepository.getAppLanguage()`/`setAppLanguage(tag)` (`null` = system) wraps it purely so ViewModels stay testable. The Settings "Sprache" section (`LanguageViewModel` + picker sheet: Systemsprache/Deutsch/English — language names are native endonyms, intentionally untranslated) applies it.
  - **`MainActivity` stays a plain `ComponentActivity`** (switching to `AppCompatActivity` would force a `Theme.AppCompat` descendant and break the Ink & Paper XML theme). Because it isn't registered with `AppCompatDelegate`, below API 33 `attachBaseContext()` wraps the base context with the stored locale manually, and the language picker calls `activity.recreate()` after a pick (both guarded by `SDK_INT < TIRAMISU`; on 33+ the platform handles both automatically).
  - **Locale-sensitive formatting**: use `ui/util/LocaleUtil.kt`'s `currentAppLocale()` / `localizedDateFormatter(pattern)` — never `Locale.getDefault()` or a hardcoded locale — because the per-app locale does not update the JVM default below API 33. **Never cache a formatter or localized string in a top-level `val`** (evaluated once at class-load, frozen on the first locale) — make it a function re-resolving per call; the same applies to composable-derived label lists. `String.format`-style calls must pass `currentAppLocale()` explicitly (`%f` renders the default locale's decimal separator); `Double.toString()`/string interpolation is locale-safe and preferred where no placeholder resource is in play.
  - **Strings in non-`@Composable` lambdas** (snackbar `launch`, `onClick`, `LaunchedEffect`): resolve via `stringResource()` into a local `val` at the top of the enclosing composable and capture it. **Strings on ViewModel state**: never build display text in the VM — expose structured data (an enum/sealed interface) and map it to a resource in the UI layer (see `BarcodeError` + `DataMessage`/`DataMessageFormat`). **No Compose context** (notifications, Glance widget): `context.getString(R.string...)`.
- **Release signing — local `keystore.properties`, CI via secrets.** The `release` signing config (`app/build.gradle.kts`) reads `storeFile`/`storePassword`/`keyAlias`/`keyPassword` from a gitignored `keystore.properties` at the project root, falling back to the env vars `RELEASE_KEYSTORE_PATH`/`RELEASE_KEYSTORE_PASSWORD`/`RELEASE_KEY_ALIAS`/`RELEASE_KEY_PASSWORD` (copy `keystore.properties.example` locally). The upload keystore (`upload-keystore.jks`, PKCS12 — **store and key password must be identical**; a PKCS12 keystore holds only one password despite `keytool` accepting two, and a mismatch fails at package time with a cryptic "Given final block not properly padded") lives outside the repo (e.g. `~/keystores/trainist/`) — **losing it means Play Store updates can never be published again under the same listing**; keep it backed up off-machine. Debug builds and unit tests need none of this. `.github/workflows/play-release.yml` (manual dispatch) builds a signed `bundleRelease` from a base64 keystore secret (`RELEASE_KEYSTORE_BASE64` + password/alias secrets) and uploads the `.aab` as a workflow artifact — it does not upload to Play Console itself (that needs a service-account setup, not yet wired).
- **App versioning — git-derived build counter.** `versionCode = git rev-list --count HEAD` (computed at configure time), `versionName = "1.0.<count>"` — nothing to bump by hand. `BuildConfig.VERSION_NAME` shows in the Settings `VersionFooter`. `scripts/git-hooks/prepare-commit-msg` stamps each commit message with the `Version: 1.0.<N>` trailer it will produce (skips if already present, so `--amend` doesn't double it); it's installed in this checkout, but **a fresh clone must copy it in manually** (`cp scripts/git-hooks/prepare-commit-msg .git/hooks/`). **CI checkouts must fetch full history** (`fetch-depth: 0`) — a shallow clone would make every CI build report `1.0.1`; the workflows all do.
